# wuminapp 多签详情页 4 bug 合并修复

## 背景

用户反馈个人多签详情页 4 个问题:
- bug 1:"该多签提案"标题改"提案列表" + 样式对齐治理多签详情 + 投票通过后卡片仍显示"投票中"(状态过期)
- bug 2:发起转账提案页"付款机构"显示"注册多签机构",应删除该字段(链端不依赖)
- bug 3:热钱包发起转账提案弹 QR 签名(应该热钱包直签,与 create page 对齐)
- bug 4:第 2、3 次创建个人多签后详情页"未找到 / 阈值 0 / 管理员 0"

## bug 4 链上事实查证

通过 9944 RPC 直接查链上 storage:

| 检查项 | 结果 | 含义 |
|---|---|---|
| `PersonalDuoqianInfo` 全部 keys | 1 条 | 链上只有 1 个个人多签 |
| `DuoqianAccounts` 全部 keys | 1 条 | 同上 |
| `PendingPersonalCreate` keys | 0 条(空) | 无待激活创建提案 |
| `VotingEngine::Proposals` keys | 3 条(1 个人 + 2 SFID 机构) | 用户的 3 次提交中**只有 1 次是个人多签** |
| `System::Account[creator]` | nonce=11, free=125 万元 | 余额绝对充足 |
| 扫所有 64 个 block 找 `ExtrinsicFailed Module(pallet=17)` | 0 命中 | 链端没有 reject 任何 propose_create_personal |

**结论**:用户"第 2、3 次"个人多签的 extrinsic **mempool 阶段被 reject 了**(SignedExtension 校验失败,如 nonce / 签名等),但 wuminapp 的 `chain_rpc.dart::submitExtrinsic` 只调 `author_submitExtrinsic`,**收到 txHash 就返回成功**,不感知 mempool reject。乐观写入 Isar 后跳详情页 → 链上无数据 → 一片错乱。

## 修复方案

### bug 4 — 提交链路 watch 化(最重要,影响所有 extrinsic)

`chain_rpc.dart::submitExtrinsic` 从 `author_submitExtrinsic` 改为 `author_submitAndWatchExtrinsic`(smoldot 已支持,verified [methods.rs:406](wuminapp/smoldot-pow/lib/src/json_rpc/methods.rs:406))。订阅 1 秒内的状态:

| TransactionStatus | 处理 |
|---|---|
| `ready` / `broadcast` / `inBlock` | 视为成功,返回 txHash |
| `invalid` / `dropped` / `usurped` / `future` | **抛错带 reject 原因** |
| 1 秒 timeout 未收到任何状态 | 乐观成功(broadcast 已发);后台继续 log |

**timeout = 1 秒**(99% mempool reject 在 100ms 内返回);GMB 链 6 分钟出块,不等 inBlock(用户体验底线)。

### bug 2 — 删除 OrgType.label 显示

`transfer_proposal_page.dart` 三处 `OrgType.label(widget.institution.orgType)` 全删:
- line 219:summary 改 `'提案转账 $amountFormatted GMB 给 $beneficiary'`(去机构前缀)
- line 228:删 `org` SignDisplayField(链端 propose_transfer Registry 不依赖此字段)

### Bug 4 设计实现要点

旧版 `submitExtrinsic` 仅调 `author_submitExtrinsic` 拿 txHash 立即返回,**不感知 mempool reject**(nonce/余额/SignedExtension 校验失败被静默忽略,Isar 写入孤立 entity)。

新版实现:
1. `SmoldotClientManager.subscribe('author_submitAndWatchExtrinsic', [hex])` 订阅
2. 监听 1 秒内的 TransactionStatus,分三类:
   - 成功(返回 txHash):`ready` / `broadcast` / `inBlock` / `finalized`
   - 失败(抛错):`invalid` / `dropped` / `usurped` / `retracted` / `future` / `finalityTimeout`
   - 仍等待(继续监听):其他
3. 1 秒 timeout 未收到任何明确状态 → 乐观成功(broadcast 已发,后台仍订阅但只 log)
4. finally 块 cancel timer + cancel subscription(防泄漏)

错误信息中文化(`_describeTxStatus`):
- invalid → "交易无效(可能 nonce 重复 / 余额不足 / 签名无效 / SignedExtension 校验失败)"
- dropped → "被交易池剔除"
- future → "nonce 大于链上已确认值"
- usurped → "被同 nonce 的另一笔交易顶替"

### Bug 2 设计要点

`OrgType.label(orgType)` 对 `OrgType.duoqian`(个人 + SFID 机构共用)输出"注册多签机构",对个人多签是误导。直接删 3 处显示,保留多签账户名作为唯一标识,链端 propose_transfer Registry 不依赖 org 字段(wumin decoder 同步删读取)。

### Bug 3 设计要点

`transfer_proposal_page._submit` 复用 [personal_duoqian_create_page.dart:228-287](wuminapp/lib/duoqian/personal/personal_duoqian_create_page.dart:228) 的成熟模式 — `wallet.isHotWallet → hotWalletManager.signWithWalletNoAuth`,`!isHotWallet → QR 签名`。多签管理员(个人 + 机构)双路径,治理机构(NRC/PRC/PRB)和区块链软件端管理员仍只支持冷钱包 — 这两条线由各自页面单独管理。

### Bug 1 设计要点

`PersonalProposalHistoryService.fetchAll` 追加 `_refreshLocalVotingEntities`:遍历 Isar 中 `status='voting'` 的 entity,挨个查链上 `Proposals[id]` 拿最新 status + tally,upsert。这步独立于 active 列表(提案终态后从 active 列表移除,但本机 entity 还停在 voting,UI 卡片显示"投票中"是假数据)。链上 storage 不存在则跳过(已被 90 天清理 → 等本机其他渠道写入历史)。
