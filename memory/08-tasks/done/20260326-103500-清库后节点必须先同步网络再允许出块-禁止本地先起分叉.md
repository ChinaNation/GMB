# 任务卡：清库后节点必须先同步网络再允许出块，禁止本地先起分叉

- 任务编号：20260326-103500
- 状态：in-progress
- 所属模块：citizenchain/node
- 当前负责人：Codex
- 创建时间：2026-03-26 10:35:00

## 任务需求

无论任何用户安装区块链软件后，节点都必须先从现有网络同步，不能因为本地是空库就自己先从创世出块，形成一条本地分叉。

## 关联背景

- 现有 `skip-empty-blocks` 逻辑为避免清库重启死锁，在 `best_number < 10` 时允许空块出块。
- 该逻辑在节点尚未接入网络时也会放行，导致清理 `db/full` 后节点可能先从创世自己出块。

## 模块边界

- 允许改动：
  - `citizenchain/node/src/service.rs`
  - `memory/01-architecture/citizenchain/CITIZENCHAIN_TECHNICAL.md`
- 不修改 runtime、node 启动协议和链上治理规则。

## 风险说明

- 如果门控过严，网络早期阶段可能出现“已连网但仍不出块”的停滞。
- 必须避免仅靠“有 bootnode 配置”就放行，因为 bootnode 只是地址，不等于已经完成网络同步。
- 需要兼容“节点临时掉线”场景，避免掉线后继续离线挖矿造成新分叉。

## 实施记录

- 已确认 `node` 启动时会注入外部 `chainspec.json`，bootnodes 并未丢失。
- 已确认当前本地先起分叉的直接原因，是 `service.rs` 中 `best_number < 10` 时无条件放行创世引导空块。
- 已在 `citizenchain/node/src/service.rs` 中把 `pool_ready` 门控改为：
  - 无同步 peer 时返回 `0`
  - `is_major_syncing()` 为真时返回 `0`
  - `best_number == 0` 时返回 `0`，要求先导入至少 1 个网络块
  - 不再保留任何“前 10 个块自动引导空块”的放行逻辑
- 验证结果：`cargo check -p node` 通过。
