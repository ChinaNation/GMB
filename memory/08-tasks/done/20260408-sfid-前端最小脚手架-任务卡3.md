# 任务卡 3:sfid 前端最小可行脚手架(hooks + api 层)

- 创建日期:2026-04-08
- 模块归属:SFID Agent(`sfid/frontend/src`)
- 依赖:任务卡 2(后端新 API + scope)
- 阻塞:任务卡 4(5 视图功能改造)
- 优先级:中

## 决策(与原方案的偏差)

原方案要求本卡做"**纯结构性 refactor**",把 4000 行 App.tsx 按 tab 拆成 `views/*/index.tsx`,
**不改功能**。但是:

1. **任务卡 4 会重写所有 tab 的 UI**(43 省卡片 / 市卡片 / 详情页 / 公权机构新 tab / 私权机构改名 / 按角色 scope 分流等)。先搬到子目录再删除,是纯浪费。
2. 大规模"纯搬运"最容易悄悄引入 bug,验证成本高、收益低。

**因此本卡压缩范围**:只建最小脚手架,App.tsx 暂不拆。任务卡 4 直接在新目录下**重写**各 tab
(搬运 + 改造一起做)。

## 目标

1. 建立 `src/hooks/` `src/views/` `src/api/` 的目录结构(空壳)
2. 抽出 3 个通用 hook:`useAuth` / `useCapabilities` / `useScope`
3. 新建 `src/api/institution.ts` — 封装任务卡 2 的新后端 API 端点
4. **App.tsx 不动**(或只动 1~2 行 import),任务卡 4 会重写

// 对应后端:

### hooks 层

**`useAuth.ts`** — 复制 App.tsx 里当前 auth state 管理(setAuth / localStorage 持久化等)的
核心逻辑,做成可复用 hook。暂时**不替换** App.tsx 里的用法,任务卡 4 再切过去。

**`useCapabilities.ts`** — 从 App.tsx 的 `capabilities` 对象派生逻辑提取:
```typescript
const cap = useCapabilities(auth);
// { canViewInstitutions, canViewMultisig, canCrudShiAdmins, ... }
```

**`useScope.ts`** — 对齐后端 `scope::rules::VisibleScope`:
```typescript
interface VisibleScope {
  provinces: string[];      // 空 = 全部
  cities: string[];         // 空 = 不限
  canWrite: boolean;
  skipProvinceList: boolean;
  skipCityList: boolean;
  lockedProvince: string | null;
  lockedCity: string | null;
}
const scope = useScope(auth);
```

## 验收

1. `npx tsc --noEmit` exit 0
2. 目录结构建好:`src/hooks/` `src/views/` `src/api/institution.ts`
3. 3 个 hook 和 1 个 api 文件都有基本实现
4. **App.tsx 字节数变化 ≤ 10 行**(本卡不动业务代码)

## 风险

- 几乎无风险。hooks 和 api 文件都是新增,不触发现有代码变更。唯一的风险是 TS 类型跟 `client.ts` 里的现有类型重复定义。
  **缓解**:`api/institution.ts` 只定义任务卡 2 新结构体,复用 `client.ts` 里的 `AdminAuth` 等现有类型。
