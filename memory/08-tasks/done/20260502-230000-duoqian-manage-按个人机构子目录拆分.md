# duoqian-manage 模块按 personal/ + institution/ 子目录完整拆分

## 背景

`duoqian-manage/src/` 已存在 `personal/` 和 `institution/` 子目录，但只是 step 1 占位（注释自承"先把目录边界建好"），全部业务代码仍堆在 `lib.rs`(3739 行)。本任务完成 step 2-10，把个人多签业务搬到 `personal/`、机构多签业务搬到 `institution/`，共用部分留 `src/` 顶层。

## 边界

- **不改 ABI**：call_index / storage 命名 / event 顺序 / error 顺序 / STORAGE_VERSION 一律不动
- **不改业务逻辑**：纯结构重构
- **不改 spec_version**
- **不动测试 mock**：保留 lib.rs 内的 `mod tests`（依赖 construct_runtime mock，跨模块拆收益小）
- **不动 voting-engine / admins-change 协议**

## FRAME 硬约束

`#[frame_support::pallet]` 宏所有 `#[pallet::*]` 注解必须留在同一个 mod pallet 内。所以拆法是：
- lib.rs 保留 mod pallet 完整骨架（Config/Storage/Event/Error/Call 签名/Hooks/Genesis/InternalVoteExecutor 入口）
- extrinsic 内部业务逻辑 delegate 到子模块的 `pub(crate) fn do_*` free fn

### 跳过的 Step 9

`activate_admin_subject` / `close_admin_subject` / `create_pending_admin_subject_for_proposal` / `remove_pending_admin_subject` 等 admin subject lifecycle helper 决定保留在 lib.rs 内 `impl<T:Config> Pallet<T>` 块。理由:
- 它们用 `Self::xxx` 直接访问 storage,搬到 free fn 形式会强制泛型化、可读性差
- 它们是 Pallet 公开 API,与 storage 紧耦合
- FRAME 风格上跨 mod 给 Pallet 加方法 vs 写 free fn 二选一,前者更地道

### ABI 兼容性

- ✓ 所有 6 个 extrinsic 的 call_index 不动(0=propose_create / 1=propose_close / 2=register_sfid_institution / 3=propose_create_personal / 4=cleanup_rejected_proposal / 5=propose_create_institution)
- ✓ 9 个 storage 命名/类型/Hasher 不动
- ✓ 13 个 event 顺序不动
- ✓ 40+ 个 error 顺序不动
- ✓ STORAGE_VERSION = 6 不变
- ✓ spec_version 不变
- ✓ 公开类型通过 `pub use` 重导出,wuminapp/wumin/sfid 端解码兼容
