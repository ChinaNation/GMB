## 实施记录

- 任务卡已创建

- 交易提交链路：`wuminapp/lib/duoqian/shared/duoqian_manage_detail_page.dart` 在 `InternalVoteService.submit` 返回 txHash 后立即写入 `PendingVoteStore` 并提示“投票已提交”，但 `wuminapp/lib/rpc/chain_rpc.dart` 的 `submitExtrinsic` 是 submit-only，返回 txHash 不代表交易已进入区块。
- “投票中”来源：`ProposalAdminVoteList` 根据 `PendingVoteStore.confirmAll` 返回的 pending pubkey 显示转圈；确认逻辑在 `wuminapp/lib/rpc/onchain.dart` 只看链上 confirmed nonce 是否大于提交时 usedNonce。若该管理员账户 nonce 没推进，就会继续显示 pending，说明交易没有被区块实际包含。
- 链上投票权限：`citizenchain/runtime/votingengine/internal-vote/src/lib.rs` 的 `do_internal_vote` 使用 `VotingEngine::AdminSnapshot` 判断管理员资格，投票成功后写 `InternalVotesByAccount` 和 `InternalTallies`。如果交易被包含但 dispatch 失败，nonce 仍会推进；当前用户描述的“无出块/一直 pending”更像交易池 future/invalid/dropped 或广播失败，而不是链上投票逻辑已成功执行。
- 进度条问题：`DuoqianManageDetailPage` 渲染 `ProposalVoteProgress` 时传入 `widget.institution.internalThreshold`。个人多签 `InstitutionInfo.orgType == OrgType.duoqian` 且没有 `internalThresholdOverride` 时，`internalThreshold` 默认为 0；创建个人多签的链上真实创建阈值是 `InternalVote::InternalThresholdSnapshot[proposal_id] = 管理员总数`。因此详情页用错阈值来源，会导致已有投票时进度条仍不按真实创建阈值变化。
- 管理员与普通阈值来源：个人多签 pending 阶段的管理员主体会写入 `AdminsChange::AdminAccounts` 且状态为 Pending，`InstitutionAdminService.fetchAdmins` 能读到管理员列表；但创建提案进度不应使用 `InstitutionInfo.internalThreshold`，应读取提案级 `InternalThresholdSnapshot` 或创建快照中的 `create_threshold`。

## 建议修复方案

- 交易确认修复：把前端文案从“投票成功”改为“投票已提交，等待出块”；为详情页增加 pending 轮询刷新，直到 nonce 推进、提案 tally 变化、链上状态终态或超时失败；交易超时/丢失时清掉 pending 记录并提示用户重新投票。
- 交易失败回显修复：把 `ChainRpc.bgWatch` 观察到的 `future/invalid/dropped/usurped` 从仅打印日志升级为可被调用方消费的状态，至少在投票详情页显示“nonce 偏大 / 余额不足 / 交易池剔除 / 被同 nonce 交易替代”等可操作提示。
- nonce 残留修复：pending 判定为 lost 或被交易池拒绝后，重置对应钱包的 `NonceManager` 本地 nonce，避免下一次继续使用偏大的 nonce 导致连续 future。
- 进度条修复：新增读取 `InternalVote::InternalThresholdSnapshot[proposal_id]` 的方法；`DuoqianManageDetailPage` 的 `ProposalVoteProgress.threshold` 改用提案级阈值。若链上快照暂时读不到，个人创建提案可回退到本地 `PersonalDuoqianProposalEntity.snapshotJson.create_threshold` 或管理员数量。
- 管理员列表一致性修复：创建/关闭提案详情页优先读取 `VotingEngine::AdminSnapshot[proposal_id][institution]` 作为投票人列表；避免用当前 `AdminsChange::AdminAccounts` 误代表提案创建时快照。

## 完整修复设计

### 问题 1：其他管理员投票后显示成功但不出块

- 修复目标：前端必须区分“已提交”“已广播”“已进块”“链上投票已生效”“交易失败/丢失”，禁止把 `txHash` 返回当作投票成功。
- 投票前校验：
  - 读取提案状态，必须是 voting。
  - 读取提案级 `AdminSnapshot`，确认当前钱包在快照内。
  - 读取 `InternalVotesByAccount`，确认当前钱包未投过票。
  - 查询 `PendingVoteStore`，确认当前钱包没有同提案 pending 记录。
  - 查询钱包余额，余额不足时直接阻止投票并提示需要手续费余额。
  - 查询 nonce 与本地 pending 状态，若存在明显 nonce future 风险，提示先等待/清理 pending。
- 提交中状态：
  - `submitExtrinsic` 返回 txHash 后只进入 `submitted`，页面文案为“投票已提交，等待出块”。
  - `author_submitAndWatchExtrinsic` 的 `ready/broadcast/inBlock/finalized/future/invalid/dropped/usurped` 必须落到业务状态。
  - `future`：提示“nonce 偏大，可能前一笔交易未确认”。
  - `invalid`：提示“交易无效，可能余额不足、签名错误、nonce 错误或无投票权限”。
  - `dropped/usurped`：提示“交易被交易池丢弃或被同 nonce 交易替代”。
- 出块后确认：
  - nonce 推进后，继续读取 `InternalVotesByAccount[proposal_id][admin]`。
  - 如果查到投票记录，才显示“投票已确认”。
  - 如果 nonce 推进但查不到投票记录，提示“交易已上链但投票未生效”，后续可扩展事件解析定位 dispatch error。
  - 如果超过超时时间 nonce 未推进，标记 lost，清除 pending 并重置该钱包本地 nonce。

### 问题 2：已有投票但进度条不变化

- 修复目标：进度条必须使用提案创建时的真实阈值和链上 tally，不得使用 `InstitutionInfo.internalThreshold`。
- 数据源调整：
  - 新增读取 `InternalVote::InternalThresholdSnapshot[proposal_id]`。
  - 新增读取 `VotingEngine::AdminSnapshot[proposal_id][institution]`，用于管理员列表和快照人数。
  - `fetchVoteTally(proposalId)` 继续读取 `InternalVote::InternalTallies`。
- 页面调整：
  - `DuoqianManageDetailPage` 增加 `_threshold` 状态字段。
  - `_load()` 并行读取 status、tally、threshold、admin snapshot、每个管理员投票记录。
  - `ProposalVoteProgress.threshold` 改为 `_threshold`。
  - 个人创建提案若暂时读不到 `InternalThresholdSnapshot`，回退到本地 snapshot 的 `create_threshold`；再不行回退到 admin snapshot 人数。
- 刷新策略：
  - 投票提交后立即显示 pending，但不增加 yes/no。
  - 轮询到链上 `InternalVotesByAccount` 后再刷新 tally 和进度条。
  - 提案 status 变为 passed/rejected/executed 时停止投票按钮和 pending 轮询。

### 验证标准

- 发起人和其他管理员都能看到同一份提案级管理员快照。
- 无手续费余额的钱包不能显示“投票成功”，必须提前提示或提交后显示明确失败。
- txHash 返回后 UI 只显示“等待出块”，不是“成功”。
- 其他管理员投票进块后，`yes/no` 和进度条同步变化。
- 提案达到创建阈值后，状态从 voting 进入 passed/executed，pending 记录被清理。
