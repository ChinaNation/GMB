# 任务卡 0.5:sfid 三角色命名全链路统一 — `super-admins/operator-admins` → `sheng-admins/shi-admins`

- 创建日期:2026-04-08
- 模块归属:SFID Agent(`sfid/backend` + `sfid/frontend` + `memory/` 文档)
- 依赖:无
- 阻塞:任务卡 1 / 2 / 3 / 4 / 5(整个 sfid 系列大重构)
- 优先级:**最高**(所有后续卡的前置)

## 背景

sfid 系统的 3 种管理员角色历史上有命名混乱:

| 角色(业务语义) | 代码目录 | Rust 模块 | 枚举值 | 序列化字符串 |
|---|---|---|---|---|
| 密钥管理员(全国 3 人) | `key-admins/` | `key_admins` | `KeyAdmin` | `KEY_ADMIN` |
| 省级管理员(每省 1 人) | **`super-admins/`** ❌ | `super_admins` | `InstitutionAdmin` | `INSTITUTION_ADMIN` |
| 市级管理员(每市 N 人) | **`operator-admins/`** ❌ | `operator_admins` | `SystemAdmin` | `SYSTEM_ADMIN` |

"super-admins"字面意思是"超级管理员",但它实际管的是**省级**机构管理员;"operator-admins"又和"系统管理员"这个中文名脱钩。
这层历史包袱在 2026-04-07 讨论文件组织时被确认为需要清理。

## 目标

把 sfid 系统的**三角色命名从目录到数据库字段值**全链路统一为以下三个名字,任何地方都不再出现旧名:

| 角色 | 目录 | 模块 | 枚举 | DB / API / TS |
|---|---|---|---|---|
| 密钥管理员 | `key-admins/` | `key_admins` | `KeyAdmin` | `KEY_ADMIN` |
| 省级管理员 | `sheng-admins/` | `sheng_admins` | `ShengAdmin` | `SHENG_ADMIN` |
| 市级管理员 | `shi-admins/` | `shi_admins` | `ShiAdmin` | `SHI_ADMIN` |

UI 中文也跟着改:
- "机构管理员" → **"省级管理员"**
- "系统管理员" → **"市级管理员"**

## 不改的地方

- ❌ **citizenchain 链端**:`institution_admins` / `snapshot_institution_admins` / `InstitutionPalletId` 等是**链上机构的管理员列表**(国储会/省储会/省储行/注册多签机构下挂的管理员公钥),跟 sfid 系统角色概念完全无关,全部保留。违反铁律 = 不改链
- ❌ **wuminapp**:`InstitutionAdminService` / `InstitutionAdminState` 是 wuminapp 查**链上** `AdminsOriginGov::CurrentAdmins` storage 的服务类,跟 sfid 角色无关,全部保留
- ❌ **历史已完成任务卡** `memory/08-tasks/done/*`:保留原文,git 历史一致性
- ❌ `key-admins/` 目录和 `KeyAdmin` 命名(已经正确)

## 风险

1. **数据库迁移不可逆**:011/012 前已有的旧数据备份后才跑,出错立刻 restore
2. **grep 漏网**:SCREAMING_SNAKE 字符串可能出现在奇怪的位置(比如日志格式字符串),验收步骤 grep 为准
3. **前后端时序**:必须"先改后端 + 跑 migration,再改前端",避免前端发旧字符串到新后端
4. **wuminapp 的 role 字段**:确认 wuminapp 登录时**不经过 sfid 后端**(它直接连链),所以不受 role 字段值变化影响。实施过程中要再次验证一次,确保没遗漏
