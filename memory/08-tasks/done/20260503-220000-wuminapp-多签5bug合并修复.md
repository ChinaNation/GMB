# wuminapp 多签详情页 5 bug 合并修复

## 背景

用户反馈 5 个问题:
- bug 1:发起创建后区块已出、其他管理员收到,但发起方按钮一直转圈
- bug 2:详情页转账卡片 + 管理员卡片高度过大,参考治理机构样式
- bug 3:提案列表方块化,参考治理机构提案事件样式
- bug 4:删账户信息中的管理员数量/通过阈值,加余额行;待激活时余额是承诺金额 + "不可用"标签
- bug 5:待激活的多签右上角不显示三个点(无 Popup 菜单);激活后才显示三个点 + "关闭多签"

## 根因 + 修复方案

### bug 1 — `await sub.cancel()` 阻塞 submitExtrinsic 返回

[chain_rpc.dart::submitExtrinsic](wuminapp/lib/rpc/chain_rpc.dart) 用 `author_submitAndWatchExtrinsic` 订阅 watch,1s timeout 后 completer.complete,但 finally 块 `await sub.cancel()` 等 native smoldot 流响应,GMB 链 6 分钟出块期持续推送 events,cancel 在某些时序下卡主流程 → submitExtrinsic 不返回 → UI 一直转圈。

**修复**:`finally` 块改 `unawaited(sub.cancel())`(已 import `dart:async`),fire-and-forget 不阻塞用户。

### bug 2 — 卡片高度对齐治理机构

参考 [institution_detail_page._buildAdminEntry:330](wuminapp/lib/citizen/institution/institution_detail_page.dart:330) 的 `InkWell + Padding(symmetric(14, 12)) + Row(36×36 icon + 标题 + chevron)` 模式,统一替换:
- `_buildAdminEntryCard`(ListTile → 自定义 Row)
- `_buildTransferEntryCard`(38×38 → 36×36;padding 16/14 → 14/12)

### bug 3 — 提案列表方块卡片

参考 [institution_detail_page._buildProposalCard:520](wuminapp/lib/citizen/institution/institution_detail_page.dart:520):每条提案是独立 Card,带 statusColor 边框 + 36×36 状态图标 + 标题/子标题 + 右侧状态徽章。替换 [personal_proposal_list_section._buildProposalTile](wuminapp/lib/duoqian/personal/personal_proposal_list_section.dart) 的 dense ListTile。

### bug 4 — 账户信息卡片改造

| 项 | 旧 | 新 |
|---|---|---|
| 名称 / SFID(机构)/ 多签地址 | ✓ | 保留 |
| **账户余额**(新增) | — | 多签地址下方,状态上方 |
| 状态 | ✓ | 保留 |
| ~~管理员数量~~ | ✓ | **删除** |
| ~~通过阈值~~ | ✓ | **删除** |

**余额取值规则**:
- `info?.status == Active` → `ChainRpc.fetchBalance(duoqianSs58)` 拿链上 free_balance(实时)
- `info?.status == Pending` → 从 Isar `PersonalDuoqianProposalEntity (action='create', status='voting')` 的 `snapshotJson.amount_fen` 取值,**显示为 "X.XX GMB"** + 灰色"不可用"标签
  (理由:链上 duoqian_account 在投票通过前 free_balance=0,但发起人已 reserve;UI 显示用户**承诺**的入金,与用户预期一致)

### bug 5 — Pending 不显示三个点

`AppBar.actions`:
- `_accountInfo?.status == Active` → 渲染 `PopupMenuButton`,只剩"关闭多签"一项(原"账户二维码"已在前一轮删除)
- 其他态(Pending / null) → `actions: []`,不渲染三个点

### Bug 1 设计要点(关键发现)

submitExtrinsic watch 版的 finally 块 **不能 await sub.cancel()**:smoldot 用 native 绑定,流的 cancel 需要等 native 端响应,GMB 链 6 分钟出块期持续推送 events,某些时序下 cancel 会卡住主流程导致 submitExtrinsic 永不返回 → UI 一直转圈。改 fire-and-forget 后 1s timeout 触发 complete 立即返回,sub 在后台异步 cancel 不阻塞用户体验。

### Bug 4 余额取值规则

| 状态 | 来源 | 标签 |
|---|---|---|
| `Active` | `ChainRpc.fetchBalance(duoqianSs58)` 链上 free_balance 实时 | 无 |
| `Pending` / `null` | 本机 Isar `PersonalDuoqianProposalEntity (action='create')` snapshot.amount_fen(发起人承诺入金) | 灰色"不可用"标签 |

理由:链上 duoqian_account 在投票通过前 free_balance=0,但发起人已 reserve 该金额;UI 显示 0 用户困惑,显示 amount 则更符合用户预期(承诺入金即将到账)。"不可用"标签明示链上未到账。

### Bug 5 设计要点

`AppBar.actions` 改为状态条件渲染 — `_accountInfo?.status == DuoqianStatus.active` 才渲染 PopupMenuButton。Pending(创建提案投票中)和 null 态(链上未找到)都不显示三个点,符合"链上未真正激活前不允许任何管理操作"的语义。原"关闭多签"对 Pending 多签会与创建提案冲突(同一 institution_id 不能并发提案)。

## 二改(2026-05-03 后续)

用户反馈 4 项继续返工(bug 3 费用问题暂不修):

### Bug 1 二改 — watch+1s timeout 仍卡,改 submit-only

第一轮把 `await sub.cancel()` 改成 `unawaited(sub.cancel())` 后 UI 仍转圈。重新分析:smoldot native binding 转发 broadcast stream 的首条 event 存在调度延迟,GMB 链 6 分钟出块的节奏下 1 秒经常拿不到 txHash → `completer.completeError('1 秒内未返回交易哈希')`,UI 反而误判失败继续转圈。

**最终方案**:`submitExtrinsic` 主流程仅调原生 `submitExtrinsicHex`(底层 `author_submitExtrinsic`),拿到 txHash 立即返回。新增 `_watchTxRejectInBackground`:fire-and-forget 8 秒内观察 invalid/dropped/usurped/future 仅打日志,不再回灌 UI。

代价:mempool reject 不在客户端拦截,排查改走 polkadot.js apps + 终端日志。决定取舍:UI 永不卡住优先。memory `feedback_extrinsic_submit_must_watch.md` 已重写为"watch 已下线"。

### Bug 2 二改 — Pending 也要显示三点(本地删除)

第一轮把 Pending 三点整体隐藏,用户反馈"不能改"。重新设计:三点永远渲染,菜单项按状态分流:
- Active → "关闭个人/机构多签"(走链上 propose_close)
- Pending/null → "删除"(走 `_confirmDeleteLocal()` 仅清本机 Isar)

`_removeFromLocal` 同时清 `personalDuoqianProposalEntitys`,否则下次返回列表会被反向索引重新拉回。

### Bug 4 二改 — 真正的 SS58 编码

第一轮 `_pubkeyToSS58` 是个误命名:实际只做 hex 截断,从未编码 SS58。改用 `Keyring().encodeAddress(bytes, 2027)` 真编码,12+8 截断适配 monospace 11 字号 ListTile。

## 三改(2026-05-03 更晚) — 反向索引重新拉回 bug

### 用户反馈

> 即使没有创建成功的多签,在手机端删除了,再次进入手机端的时候又会从节点下载到手机端。

### 根因

二改的 `_confirmDeleteLocal` 仅删本地 Isar entity。但:

1. **链上侧**:[duoqian-manage/src/personal/create.rs:135-144](citizenchain/runtime/transaction/duoqian-manage/src/personal/create.rs:135) `do_propose_create_personal` **在 propose 阶段就**调 `create_pending_admin_account_for_proposal`,把 admin 列表写进 `admins-change::Institutions`(以 institution_id 为 key)。
2. **客户端反向索引**:[duoqian_discovery_service.dart:117-201](wuminapp/lib/duoqian/shared/duoqian_discovery_service.dart:117) `discoverByMyWallets` 扫的就是 `admins-change::Institutions`,**不区分 status=Pending/Active**,只要 admin 列表有本钱包就 upsert PersonalDuoqianEntity。
3. 用户删本地 → 30 分钟节流过期 / 下次进列表 → 反向索引重新插回 `discoveredViaAdmin=true` 的 entity。

### 设计决定(用户敲定)

> 还没有激活的多签,所有管理员都可以删除,这个删除就是本地和链上一起删除,这样所有管理员的手机上就不会再有这个多签出现了

= **撤销创建走链上反对投票**(无需 spec_version+1):

- 链上 [voting_engine/internal_vote.rs:482-485](citizenchain/runtime/governance/voting-engine/src/internal_vote.rs:482) **已有早期否决**:`tally.yes + remaining < threshold` 立即 STATUS_REJECTED。
- 个人多签创建 threshold=全员 → **任意一票反对即 REJECTED**。
- [duoqian-manage/src/lib.rs:1097-1107](citizenchain/runtime/transaction/duoqian-manage/src/lib.rs:1097) `on_internal_vote_finalized(approved=false)` 自动调 [cleanup_pending_personal_create](citizenchain/runtime/transaction/duoqian-manage/src/personal/execute.rs:26):unreserve + 删 `DuoqianAccounts` / `PersonalDuoqianInfo` / `PendingPersonalCreate` / **`admins-change::Institutions`**。
- 反向索引下次扫不到该 institution_id → 所有管理员设备孤立 entity 自动清理(`discoverByMyWallets` Step 6 reverse-validate 已实现)。
- 链上 0 改动,纯客户端整改。

### 改动

| 文件 | 改动 |
|---|---|
| `lib/duoqian/shared/duoqian_account_info_page.dart` | 1) 新增 `_confirmRevokeCreate`:取本机热钱包 admin → `PersonalPendingCreateLookup.findActiveCreate` → `InternalVoteService.submit(approve=false)` + `_removeFromLocal` + Navigator.pop。2) `_openClosePage` 提交成功后**不再立即删本地**(propose_close 通过前 admins-change 还在,反向索引会拉回)。3) PopupMenu 三态分流:个人 Active "关闭" / 个人 Pending "撤销创建" / 机构 Active "关闭" / 机构 Pending **不展示菜单**(交 SFID 治理流程)。4) 新增 `_shouldShowMenu` helper |

### 已知限制

- **冷钱包用户**:`_confirmRevokeCreate` 仅支持热钱包,冷钱包持有者点撤销会得到 SnackBar 提示"请到管理员列表扫码投反对票"——此时走 PersonalAdminListPage 的现有反对投票路径,语义等价。
- **机构 Pending 撤销**:不支持。机构创建是 SFID 治理流程,wuminapp 不应作为撤销入口。机构详情页 Pending 状态右上角不展示菜单。
- **本钱包无 admin**:撤销按钮提示"请先导入此多签的管理员钱包"。
- **链上提案已被并发处理**:撤销提交瞬间正好被最后一票通过 → 链上拒绝(AlreadyVoted/ProposalNotFound),UI 弹"撤销失败:..."snackbar,用户可手动刷新详情页。
