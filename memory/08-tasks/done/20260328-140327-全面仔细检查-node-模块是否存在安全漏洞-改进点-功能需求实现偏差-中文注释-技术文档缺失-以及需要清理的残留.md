## 审查结论

- 发现 2 个高优先级安全问题，分别是“高权限代签 RPC 无鉴权”和“PoW 作者身份只靠未认证 digest 约定”。
- 还存在 2 个明显工程/功能问题：`fee_blockFees` 统计口径错误、`gpu-mining` feature 当前无法编译。
- 中文注释总体够用，但 `node` 缺少独立中文技术文档；测试覆盖也明显不足。

## 发现问题

1. `reward_bindWallet` / `reward_rebindWallet` 是无鉴权的代签 RPC。
   - 证据：`node/src/rpc.rs` 直接注册这两个方法，收到 RPC 请求后就从本地 keystore 取 `powr` 密钥签名并提交交易，没有任何口令、来源、会话或本地交互确认。
   - 风险：任何能访问该 RPC 的调用方都能替节点改奖励钱包。
   - 放大条件：当前 `nodeui` 启动节点时实际使用了 `--unsafe-rpc-external --rpc-methods Unsafe --rpc-cors all`，会把这类高权限 RPC 暴露到外部。

2. PoW 作者身份没有密码学绑定，协议只要求 pre-runtime digest “能解码成账户”。
   - 证据：`node/src/service.rs` 的 `SimplePow::verify` 只校验 `pre_digest` 可解码为 `AccountId`；runtime 的 `PowDigestAuthor` 也只是再次直接解码。
   - 影响：奖励归属、手续费分账和矿工身份展示依赖“诚实 node 会把本地 keystore 公钥写进 digest”的实现约定，而不是链上协议保证。

3. `fee_blockFees` 会少算带 tip 交易的真实手续费。
   - 证据：`node/src/rpc.rs` 通过累加 `OnchainTransactionPow::FeePaid.fee` 统计手续费；但 runtime 中该事件只写 `base_fee`，明确不含 `tip`。

4. `gpu-mining` 功能当前并未真正实现完成。
   - 证据：`cargo check -p node --features gpu-mining` 失败，`node/src/gpu_miner.rs` 存在 OCL API 调用不匹配和 `U256::to_big_endian` 调用错误。

5. `node/src` 自定义逻辑几乎没有测试覆盖。
   - 证据：`cargo test -p node` 通过，但实际运行结果是 `0 tests`；当前没有针对 `reward_* RPC`、`fee_blockFees`、`sync_state_genSyncSpec`、挖矿门控的回归测试。

6. 技术文档不完整。
   - 证据：`memory/05-modules/citizenchain/` 下没有 `node` 模块的独立中文技术文档；当前只有其他模块文档零散引用 `node/src/service.rs`、`node/src/rpc.rs`。

7. 有清理残留。
   - `cargo fmt -p node -p sc-consensus-grandpa -- --check` 失败，`node/vendor/src/lib.rs` 有格式差异。
   - `node/vendor` 里仍保留多处上游 `TODO/FIXME` 注释，说明本地覆盖 GRANDPA 代码还没有完全收口。
