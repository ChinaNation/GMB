# 客户端接入双层 ID 与反向索引(PR-Z)

- **日期**: 2026-05-04
- **模块**: Mobile Agent + Blockchain Agent — `wuminapp` + `citizenchain/node`
- **优先级**: 高
- **依赖**: PR-X(20260504-100000) + PR-Y(20260504-130000) 已完成

## 背景

PR-Y 落地链上双层 ID + 反向索引,客户端必须同步:
1. **删除 `id ~/ 1_000_000` 类硬编码** — 改读链上 `ProposalDisplayId`
2. **改用反向索引迭代** — 不再"全量扫主键 + 客户端事后过滤"
3. **顺手修 `_hasMore` 翻页 bug** — 反向索引模式下 cursor 翻页天然不会卡空页

链上 storage prefix `'VotingEngine'` 不动,无需改 `_twoxx128String('VotingEngine')` 字面量。
wumin 冷钱包 / sfid 0 改动。

## 决策(已与 user 拍板)

1. **展示号格式**:`2026000123`(年份 + 6 位补零年内序号,无分隔符;seq 突破 6 位时自动扩展到 7、8 位)
2. **列表数据源**:
   - 机构详情页 → `ProposalsByInstitution[sfidNumber]`(只本机构)
   - 公民-治理页 → `ProposalsByOrg[NRC] ∪ ByOrg[PRC] ∪ ByOrg[PRB]`(三 org 并集,排除 ORG_DUOQIAN)
3. **不做** `list_proposals_by_year` 归档视图(等产品需要再加)
4. **展示号仅内存缓存**,App 启动时一次 batch 拉取,不写 Isar

## 范围与改动量

| 模块 | 改动 | 估算行数 |
|---|---|---|
| wuminapp | 接入反向索引 + 删硬编码 + 修 `_hasMore` bug | ~570 |
| citizenchain/node 后端(Rust) | async `format_proposal_id` + 4 个反向索引 Tauri 命令 | ~150 |
| citizenchain/node 前端(TS) | api.ts wrappers + 各组件 `formatProposalId` 适配 | ~100 |
| wumin 冷钱包 | **0 改动** | 0 |
| sfid | **0 改动** | 0 |
| 测试更新 | 全栈 | ~200 |

### Phase Z-2:citizenchain/node Tauri(Blockchain Agent)

#### Z-2.1 — `node/src/governance/proposal.rs` 改造

```rust
// 旧
fn format_proposal_id(id: u64) -> String { /* id / 1_000_000 计算年份 */ }

// 新
async fn format_proposal_id(api: &ApiClient, id: u64) -> Result<String> {
    let display = api.fetch_proposal_display_id(id).await?
        .ok_or_else(|| anyhow!("display meta missing for {id}"))?;
    Ok(format!("{}{:06}", display.year, display.seq_in_year))
}
```

## 风险点

| 风险 | 缓解 |
|---|---|
| wuminapp `ProposalCache` 升级后旧缓存键混淆 | App 启动一次性 `ProposalCache.clear()`,反正只是内存 |
| 列表页一次拉太多 ProposalDisplayId,smoldot 慢 | 用 `state_getStorageBatch`,开发期数据量小 |
| 反向索引迭代取到的 ID 没有 ProposalDisplayId(理论不可能) | UI fallback 显示原始 u64,不阻塞列表 |
| 隐藏的 `~/ 1_000_000` 漏修 | 改完后用 `grep -rn "1_000_000\|1000000" lib/` 扫一遍 |
| Tauri 命令签名变化破坏前端 | 每次改后端 invoke 签名后立刻同步前端 invoke wrapper |
