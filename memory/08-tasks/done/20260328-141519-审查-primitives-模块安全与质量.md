## 主要发现

1. `pow_const.rs` 同时保留了两套互相冲突的时间口径：`BLOCKS_PER_DAY` 仍基于创世期 30 秒出块推导，而 `BLOCKS_PER_YEAR` 已单独固定为 87,600。`count_const.rs` 的投票期限继续依赖 `BLOCKS_PER_DAY`，会让“30天投票期限”在运行期 6 分钟出块口径下膨胀到约 360 天；若继续坚持“空块不允许上链”，实际墙钟时间还会更长。
2. `primitives` 作为 runtime / node / 脚本共享根模块，当前只有目录级 README，没有像其他关键模块那样的技术说明文档，缺少对时间常量、地址表来源、聚合保留地址规则、测试约束和跨模块影响面的系统说明。
3. `cargo test -p primitives` 当前只覆盖 `china_ch` 的人口总数和质押总量两条校验，没有覆盖：
   - `china_zb` 保留地址表是否排序、去重、与源数据一致
   - `china_cb` / `china_ch` 的固定长度编码 helper
   - `count_const` 中治理阈值与机构数量的联动
   - `genesis.rs`、`citizen_const.rs`、`pow_const.rs` 之间的交叉不变量
4. 低优先级清理项：
   - `china_jc.rs` / `china_jy.rs` / `china_lf.rs` / `china_sf.rs` / `china_zf.rs` 中都定义了未被引用的 `EMPTY_DUOQIAN_ADMINS`
   - `count_const.rs` 里 `RESOLUTION_ISSUANCE_MAX_ALLOCATIONS = PRB_COUNT` 的注释写成了“与省储会数量一致”，和实际常量不符
