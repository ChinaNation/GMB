# 任务卡 2:机构/账户两层数据模型 + 权限过滤 + 模块化重构

- 创建日期:2026-04-08
- 模块归属:SFID Agent(`sfid/backend`)
- 依赖:任务卡 0.5(三角色命名)+ 任务卡 1(sfid 工具归位)
- 阻塞:任务卡 3(前端 refactor)/ 4(5 视图) / 5(机构详情页)
- 优先级:高

## 背景

任务卡 1 完成了 sfid 工具模块的统一入口,并在 `sfid/category.rs` 预埋了 `InstitutionCategory` 枚举。
本任务卡落地两件事:

1. **把混在一起的 "机构" 和 "账户" 拆成两层模型**,对齐链上 `SfidRegisteredAddress::<T>(sfid_id, name) → duoqian_address` 的 DoubleMap 语义
2. **引入 `scope/` 模块**,所有 list API 按登录管理员的角色/省/市自动过滤

## 背景知识

### 链端派生协议(参见 `citizenchain/runtime/transaction/duoqian-manage-pow/src/lib.rs:1135`)

```rust
derive_duoqian_address = BlakeTwo256(
    b"DUOQIAN_SFID_V1"
    || ss58_prefix_le_bytes
    || sfid_id_bytes
    || name_bytes     // ← 这里的 name 是"账户名称",不是"机构名称"
)
```

**关键**:链只认两个维度 `(sfid_id, name)`。一个 `sfid_id` 挂多个 `name` → 派生多个 `duoqian_address`。

### 当前数据模型问题

现有 `MultisigSfidRecord`(`models/mod.rs`)把"机构"和"账户"塞在一起:

- `site_sfid` 既当机构号又当记录主键
- `institution_name` 实际被当成"账户名称"发上链
- 不支持"一个机构多个账户"

## 目标

1. 数据模型分两层:`MultisigInstitution`(每个 sfid_id 唯一)+ `MultisigAccount`(sfid_id + account_name 复合 key)
2. 机构分类持久化:`category: InstitutionCategory` 字段显式存 `public_security/gov_institution/private_institution`
3. API 拆两步:`institution/create`(只生成 sfid_id,不上链) + `institution/{sfid_id}/account/create`(上链 register_sfid_institution)
4. 所有 list API 按 scope 过滤:KEY_ADMIN 全国 / SHENG_ADMIN 本省 / SHI_ADMIN 本市
5. 同一 sfid_id 下 account_name 唯一性硬校验(避免重复推链白交手续费)
6. 数据迁移:老 `multisig_sfid_records` 6 条失败记录 + 1 条成功记录拆成机构 + 默认账户

### 数据模型(institutions/model.rs)

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MultisigInstitution {
    pub sfid_id: String,                  // 主键,参与派生
    pub institution_name: String,         // "广州市公安局" 等,sfid 系统展示用,不进链
    pub category: InstitutionCategory,    // PublicSecurity / GovInstitution / PrivateInstitution
    pub a3: String,                       // GFR / SFR / FFR
    pub p1: String,                       // "0" / "1"
    pub province: String,
    pub city: String,
    pub institution_code: String,         // ZF / LF / SF / ...
    pub province_code: String,            // r5 前 2 字符
    pub created_by: String,
    pub created_at: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MultisigAccount {
    pub sfid_id: String,                  // 外键 → MultisigInstitution
    pub account_name: String,             // 账户名,参与派生,同 sfid 下唯一
    pub duoqian_address: Option<String>,  // 链上派生地址 (hex, no 0x)
    pub chain_status: MultisigChainStatus,// Pending / Registered / Failed
    pub chain_tx_hash: Option<String>,
    pub chain_block_number: Option<u64>,
    pub created_by: String,
    pub created_at: DateTime<Utc>,
}

pub type AccountKey = (String, String); // (sfid_id, account_name)
```

### Store 层(runtime_cache_entries)

两个新 cache key:

- `multisig_institutions` → `HashMap<sfid_id, MultisigInstitution>`
- `multisig_accounts` → `HashMap<(sfid_id, account_name), MultisigAccount>`

保留 `multisig_sfid_records` cache key 一段时间,只读不写,作为**回读兜底**,便于回滚。下一张任务卡(或 3 天后)再删除。

### API 端点

| 方法 | 路径 | 功能 |
|---|---|---|
| POST | `/api/v1/institution/create` | 生成 sfid_id + 写机构,**不上链** |
| POST | `/api/v1/institution/:sfid_id/account/create` | 创建账户并上链 `register_sfid_institution` |
| GET | `/api/v1/institution/list?category=...&province=...&city=...` | 按 scope 过滤的机构列表 |
| GET | `/api/v1/institution/:sfid_id` | 机构详情(含基本信息)|
| GET | `/api/v1/institution/:sfid_id/accounts` | 机构下所有账户 |
| DELETE | `/api/v1/institution/:sfid_id/account/:account_name` | 删除账户(仅软删/标记,不触链)|

**暂保留 legacy 路由不动**(`/api/v1/admin/multisig-sfids*`,`/api/v1/admin/cpms/*`),作为前端过渡期。

### 权限规则(scope)

```rust
pub struct VisibleScope {
    pub provinces: Vec<String>,  // 空 = 全部
    pub cities: Vec<String>,     // 空 = 不限市
    pub can_write: bool,
    pub skip_province_list: bool,
    pub skip_city_list: bool,
    pub locked_province: Option<String>,
    pub locked_city: Option<String>,
}

pub fn get_visible_scope(auth: &AdminAuth) -> VisibleScope {
    match auth.role {
        KeyAdmin    => { /* 全部,can_write=true */ }
        ShengAdmin  => { /* 本省,skip_province_list=true */ }
        ShiAdmin    => { /* 本省+本市,skip_province_list+skip_city_list=true */ }
    }
}
```

所有 list API 在 handler 入口调 `filter_by_scope(rows, &scope)`。

### 同 sfid 下 account_name 唯一性

`service::ensure_unique_account_name(sfid_id, account_name, store)` 在 `account/create` 前调用,
发现已存在直接返回 409 CONFLICT,避免白交链上手续费。

### 审计日志新 action

- `INSTITUTION_CREATE` — 生成机构 sfid_id(不上链)
- `ACCOUNT_CREATE_SUBMIT` — 提交上链 register_sfid_institution
- `ACCOUNT_CREATE_CHAIN_OK` — 链上成功
- `ACCOUNT_CREATE_CHAIN_FAIL` — 链上失败
- `ACCOUNT_DELETE` — 软删除账户

旧 `MULTISIG_SFID_GENERATE` / `MULTISIG_SFID_CHAIN_REGISTER` 保留,老路由继续用。

### 数据库迁移 `014_multisig_two_layer.sql`

**只动 runtime_cache_entries 里的缓存 payload**,表结构不改(两层模型存在 json payload 里)。
- 读 `multisig_sfid_records` payload
- 对每条记录:
  - 创建 `MultisigInstitution`(category 按 a3+code+name 分类)
  - 创建一个 `MultisigAccount`,account_name 默认为 `"默认账户"`,chain_status 沿用原记录
- 写入 `multisig_institutions` 和 `multisig_accounts` 两个新 cache key
- **不删除**老 `multisig_sfid_records`(兜底)

## 验收

1. `cargo build` 通过,warning 数 ≤ baseline(35)+ 少量新增(预期 ≤ 10,且只能是 dead_code 不可以是 unused_imports)
2. `cargo test` 通过
3. migration 014 执行成功,`runtime_cache_entries` 多出 `multisig_institutions` 和 `multisig_accounts` 两个 entry
4. curl 测三角色:
   - KEY_ADMIN `GET /institution/list` 返回全部
   - SHENG_ADMIN `GET /institution/list` 只返回本省
   - SHI_ADMIN `GET /institution/list` 只返回本市
5. 新建机构 + 新建账户两步流程可完整走通
6. 同 sfid 下两次 create 同名账户 → 第二次返回 409
7. 老路由 `/api/v1/admin/multisig-sfids*` 仍然工作(向后兼容)

## 风险

1. **Store 层动作大**:双写 multisig_sfid_records + 两层新结构,数据可能不一致。策略:新建 API 只写新结构,老 API 只读不写老结构,中间加一层 sync_legacy 做迁移
2. **scope 过滤可能误伤现有页面**:老的 `list_multisig_sfids` 如果直接套上 scope 过滤,可能导致 KEY_ADMIN 看不到旧数据。策略:老路由保持现行行为不变,只给新路由加 scope
3. **迁移不可逆**:但老缓存键保留,可从老键恢复
4. **审计日志格式不回归**:新 action 只在新 API 里写,老 API 继续写老 action
5. **文件长度爆炸**:institutions/ 下 6 个文件平均可能接近 300 行,需要严格自律按职责切分
