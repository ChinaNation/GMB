# 投票引擎与机构管理模块重构(Phase 1 / 0 行为变化)

- **日期**: 2026-05-05
- **模块**: Blockchain Agent + Mobile Agent — `citizenchain/runtime/votingengine` + `citizenchain/runtime/governance/org-manage`(原 `transaction/duoqian-manage`)
- **优先级**: 高
- **依赖**: PR-X(votingengine 提级 + 拆分)、PR-Y(双层 ID 代码就绪未激活)、PR-Z(客户端反向索引接入)已完成

## 背景

当前 votingengine 把内部投票 / 联合投票 / 老公民投票(实质是联合公投)三种模式塞在 `vote/` 单目录,概念上不对称。同时 `duoqian-manage` 的"多签账户管理"实际是机构生命周期的统一入口,将来要承载公权机构 / 其他机构的注册,需要语义扩展。

本次**只做命名 + 目录重构**,与 PR-X(votingengine 改名)同样的模式:链上 alias / storage prefix / MODULE_TAG / extrinsic 名全部保留,**0 行为变化,spec_version 维持 0**。

公权 / 其他机构的注册路径(Phase 2)和公民投票新模式(Phase 3)留给后续任务卡。

## 决策(已与 user 拍板)

1. **目录重整**:`votingengine/src/vote/` 拆为 3 个平级目录
   - `internal/` — 内部投票(原 `vote/internal.rs`)
   - `joint/` — 联合投票(含联合公投兜底);包含 `joint.rs`(原 `vote/joint.rs`)+ `jointreferendum.rs`(原 `vote/citizen.rs` 改名)
   - `citizen/` — 全新公民投票(本卡只建空骨架,Phase 3 实现)
2. **机构管理改名**:`transaction/duoqian-manage/` → `governance/org-manage/`
3. **链上身份保留**:`construct_runtime!` alias `DuoqianManage` 不变,storage prefix 不变,MODULE_TAG `b"dq-mgmt"` 不变,extrinsic 名 `citizen_vote` 不变
4. **ORG 常量重排**(Rust 层标识符):`ORG_DUOQIAN(=3)` 改名 `ORG_REN(=3)` 同值;新增 `ORG_PUP=4` / `ORG_OTH=5`(Phase 2 才启用业务路径)
5. **命名风格**:位置决定风格,顶级 `runtime/X/` 单词无分隔符(`votingengine`),子级 `runtime/governance/X/` kebab-case 与 `admins-change` 等邻居一致(`org-manage`),Rust 模块标识符 `votingengine` / `org_manage` / `jointreferendum` / `citizen`

## 范围与不做的事

### 范围外(留给后续任务)

- ❌ election 公民投票实现(Phase 3)
- ❌ wuminapp / Tauri UI 拆分(Phase 4)

## 实现步骤

### Step 1.A votingengine 目录重整

```
git mv votingengine/src/vote/internal.rs       votingengine/src/internal/mod.rs
git mv votingengine/src/vote/joint.rs          votingengine/src/joint/joint.rs
git mv votingengine/src/vote/citizen.rs        votingengine/src/joint/jointreferendum.rs
新建 joint/mod.rs  ── 声明 mod joint; mod jointreferendum;
新建 citizen/mod.rs + ballot.rs + candidates.rs + tally.rs + rules.rs + scope.rs(全空 stub)
git rm vote/mod.rs(目录消失)
```

`lib.rs` 改:
- 删:`pub mod vote;`
- 加:`pub mod internal; pub mod joint; pub mod citizen;`
- 调整 `pub use`:
  - `pub use vote::citizen::{...}` → `pub use joint::jointreferendum::{...}`
  - `pub use vote::internal::ORG_DUOQIAN` → `pub use internal::ORG_REN`

全栈替换:
- `crate::vote::internal::` → `crate::internal::`
- `crate::vote::joint::` → `crate::joint::joint::`
- `crate::vote::citizen::` → `crate::joint::jointreferendum::`
- 外部 pallet 同步:`votingengine::vote::internal::` → `votingengine::internal::` 等

### Step 1.B jointreferendum 内部函数改名

`joint/jointreferendum.rs` 内部 `citizen_*` 系列 helper 函数改名 `jointreferendum_*`,但 **`pub fn citizen_vote` extrinsic 名保留不动**(动了就要 spec_version bump)。

涉及修改:
- 内部 helper(私有 fn / pub(crate) fn)
- 测试代码引用
- 注释

### Step 1.C ORG 常量重排

`votingengine/src/internal/mod.rs`:
```rust
pub const ORG_NRC: u8 = 0;
pub const ORG_PRC: u8 = 1;
pub const ORG_PRB: u8 = 2;
pub const ORG_REN: u8 = 3;   // 原 ORG_DUOQIAN
pub const ORG_PUP: u8 = 4;   // 新增,Phase 2 启用
pub const ORG_OTH: u8 = 5;   // 新增,Phase 2 启用
```

`is_valid_org`:本阶段只接受 0~3(REN 是 DUOQIAN 改名),PUP/OTH 占位但 Phase 2 才启用业务路径(本阶段写 `pub fn is_valid_org(org: u8) -> bool { matches!(org, 0..=5) }` 也行,因为 Phase 1 不会有任何 PUP/OTH 提案被实际创建)。

全栈替换:
- Rust:`ORG_DUOQIAN` → `ORG_REN`(本 crate 内 + 外部 pallet)
- Dart:wuminapp `ORG_DUOQIAN` 常量 → `ORG_REN`(只 Dart 标识符,不影响链上)

### Step 1.D duoqian-manage → org-manage

```
git mv citizenchain/runtime/transaction/duoqian-manage  citizenchain/runtime/governance/org-manage
```

改造点:
- `org-manage/Cargo.toml`:`name = "duoqian-manage"` → `name = "org-manage"`
- 7 个依赖的 Cargo.toml `path` 字段
- workspace 顶层 `runtime/Cargo.toml` + `citizenchain/Cargo.toml`
- 全栈 `use duoqian_manage::` → `use org_manage::`(424+ 处);crate 在 Cargo 是 `org-manage`,Rust import 路径 `org_manage`
- `runtime/src/lib.rs` `construct_runtime!`:
  ```rust
  pub type DuoqianManage = org_manage;   // ← alias 保留 DuoqianManage,crate Cargo 名 org-manage,Rust 路径 org_manage
  ```

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| extrinsic `citizen_vote` 名保留但内部代码挪到 jointreferendum,语义错位 | `pub fn citizen_vote` 加注释解释:`citizen_vote 是历史保留名,实质实现联合公投(joint referendum)` |
| `crate::vote::*` 引用全栈散落,漏改 | 依赖 `cargo check --workspace`,任何遗漏立即编译失败 |
| `ORG_DUOQIAN` 在 Dart 客户端漏改 | flutter analyze + grep 双保险 |
| Phase 2/3 规划变化导致 ORG_PUP/OTH 编码不再合适 | 占位编号但 `is_valid_org` 不放行任何 PUP/OTH 业务调用,Phase 2 启动前可调整 |
