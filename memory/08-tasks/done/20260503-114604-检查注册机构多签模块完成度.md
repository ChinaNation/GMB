## 完成度结论

总体判断：注册机构多签不能标记为完成。链上核心和 SFID 注册信息凭证已经基本成型，但产品端到端闭环仍未打通，尤其是 wuminapp 调用口径、wumin decoder 口径、节点前端扫码提交闭环存在阻断。

分层完成度：

- 链上 `duoqian-manage` 机构创建核心：约 85%。`register_sfid_institution` 与 `propose_create_institution(call_index=5)` 已落地，10 字段注册 payload 与 SFID `registration-info` 对齐，创建提案 reserve / pending / 激活 / 拒绝清理路径有单测覆盖。
- `sfid-system` 省管理员双层签名基础：约 85%。`ShengAdmins` / `ShengSigningPubkey` 双层模型、unsigned extrinsic 和测试已落地。
- SFID 后端机构注册信息接口：约 80%。`/api/v1/app/institutions/:sfid_number/registration-info` 已返回 `sfid_number / institution_name / account_names[]` 与 `credential.province / signer_admin_pubkey / signature`，但移动端仍没有按该接口接入机构创建。
- 节点端清算行注册机构多签：约 55%。Rust Tauri 后端能构造 10 字段 `propose_create_institution` call_data，前端能拉 registration-info，但创建页仍以 alert 占位提示等待扫码握手，真实 submit 路径未接上。
- wumin 冷钱包 decoder：约 55%。已有 `pallet=17 call=5` decoder 测试通过，但当前 decoder/测试仍按旧 14 字段口径解析 `subject_property / sub_type / parent_sfid_number`，与 runtime 已收口的 10 字段不一致。
- wuminapp 机构多签入口：约 20%。正式提交函数仍使用废弃 `call_index=0` 单账户机构创建口径；账户查询 API 路径使用单数 `/api/v1/app/institution/...`，与 SFID 后端复数路由不一致；局部测试当前编译失败。
- 文档与残留：约 65%。`DUOQIAN_TECHNICAL.md` 与 SFID institutions 文档已记录 10 字段方向，但 wumin / wuminapp 侧仍有旧字段残留与失败测试。

## 关键证据

- runtime 当前真实入口：`citizenchain/runtime/transaction/duoqian-manage/src/lib.rs` 中 `propose_create_institution` 为 `call_index=5`，参数收口为 `sfid_number / institution_name / accounts / admin_count / duoqian_admins / threshold / register_nonce / signature / province / signer_admin_pubkey`。
- runtime 机构创建实现：`citizenchain/runtime/transaction/duoqian-manage/src/institution/create.rs` 已做省管理员双层验签、账户校验、reserve、pending 机构和全员创建投票。
- SFID 注册信息接口：`sfid/backend/institutions/chain_duoqian_info.rs` 已实现 `/registration-info`，按稳定账户名顺序签发凭证并返回 `signer_admin_pubkey`。
- 节点端 call_data：`citizenchain/node/src/offchain/duoqian_manage/signing.rs` 明确按 10 字段编码。
- 节点前端未闭环：`citizenchain/node/frontend/offchain/duoqian-manage/create-multisig.tsx` 仍用 alert 提示 `[follow-up] 待接入冷钱包扫码握手`，真实 `submitProposeCreateInstitution` 被注释。
- wumin decoder 残留：`wumin/lib/signer/payload_decoder.dart` 和 `wumin/lib/signer/pallet_registry.dart` 仍描述/解析 `subject_property / sub_type / parent_sfid_number`，与当前 runtime 10 字段冲突。
- wuminapp 阻断：`wuminapp/lib/duoqian/shared/duoqian_manage_service.dart` 的 `submitProposeCreateInstitution` 仍写 `call_index=0`，不是 `call_index=5`；`wuminapp/lib/wallet/capabilities/api_client.dart` 请求 `/api/v1/app/institution/:sfid_number/accounts`，但 SFID 后端路由是 `/api/v1/app/institutions/:sfid_number/accounts`。

## 主要缺口

1. wuminapp 正式机构多签创建仍调用废弃 `call_index=0`，链上该入口已删除留洞，真实提交必失败。
2. wuminapp 账户查询路径单复数错误，无法命中 SFID 当前公开账户列表路由。
3. wuminapp 机构创建没有接入 SFID `/registration-info`，缺少真实 `register_nonce / signature / province / signer_admin_pubkey / account_names[]` 数据源。
4. wumin 冷钱包 decoder 仍按旧 14 字段解析 `propose_create_institution`，当前节点端生成的 10 字段 call_data 会解码失败或落入黄牌。
5. 节点前端创建机构多签只构造 QR 请求，不完成扫码回执提交和链上交易提交。
6. wuminapp 相关测试与实现不同步，测试文件引用不存在的服务函数和 DTO。

## 建议修复顺序

1. 先修 wumin decoder：把 `propose_create_institution(17.5)` 改为 10 字段，删除 `subject_property / sub_type / parent_sfid_number` 解析和测试残留。
2. 再修 wuminapp：账户查询路由改复数，新增拉取 `/registration-info`，`submitProposeCreateInstitution` 切到 call_index=5/10 字段并补齐字节一致性测试。
3. 然后修节点前端：复用现有扫码签名 flow，去掉 alert 占位，接上 `submitProposeCreateInstitution`。
4. 最后跑跨端验收：SFID 创建机构和账户 → 拉 registration-info → wumin/wuminapp/节点端构造同一 call_data → 冷钱包解码显示绿牌 → 回签 → 上链 pending → 投票通过 → 机构账户 Active。
