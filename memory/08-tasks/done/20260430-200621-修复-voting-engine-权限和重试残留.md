# 修复 voting-engine 权限和重试残留

## 任务需求

修复上一轮 review 发现的 voting-engine 统一状态机残留问题：

- `update_proposal_data` / `store_proposal_object_for` / `remove_proposal_object` 仍可能被 runtime 内其他 pallet 伪造 `module_tag` 后跨模块写入或删除。
- `can_cancel_passed_proposal` 默认放行，导致管理员可提前终止多数自动执行失败后的 `PASSED` 重试提案。
- 自动执行返回 `FatalFailed` 时没有走统一 `EXECUTION_FAILED` 终态清理 hook。
- 检查并清理 joint proposal 可返回 `RetryableFailed` 但缺少手动 retry/cancel 策略的边界。

## 影响范围

- `citizenchain/runtime/governance/voting-engine`
- `citizenchain/runtime/governance/runtime-upgrade`
- 使用 `InternalVoteResultCallback` / `JointVoteResultCallback` 的业务模块
- voting-engine / runtime-upgrade 技术文档

## 修复目标

- 业务模块不能通过 caller-supplied `module_tag` 直接改写 voting-engine 存储。
- 取消 `PASSED` 重试提案必须由 owner 模块显式允许，默认拒绝。
- 所有进入 `STATUS_EXECUTION_FAILED` 的路径统一触发终态清理 hook。
- joint proposal 不允许进入当前只支持 internal 的 retry state。

## 验收项

- 编译和相关单测通过。
- 文档同步更新。
- `rg` 检查无旧公开写入口和旧语义残留。

## 完成记录

- 新增 `ProposalCancelDecision`，`can_cancel_passed_proposal` 默认返回 `Ignored`，只有 owner 模块显式 `Allow` 才允许管理员取消 `PASSED` 重试提案。
- 删除生产可用的 `update_proposal_data` / `store_proposal_object_for` 写入口，`remove_proposal_object` 仅保留测试作用域；runtime-upgrade 改为创建时一次性写入 data + object。
- 自动 `FatalFailed`、手动重试耗尽、retry deadline、cancel 等执行失败终态统一触发 `on_execution_failed_terminal` 和终态清理。
- joint callback 若误返回 `RetryableFailed`，投票引擎兜底进入 `STATUS_EXECUTION_FAILED`，避免进入 internal-only retry state。
- 更新 runtime-upgrade 的状态语义：业务摘要不再回写 `ProposalStatus`，真实终态以 voting-engine `Proposal.status` 为准。
