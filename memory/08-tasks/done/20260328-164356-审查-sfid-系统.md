## 待确认问题

- `sfid/backend` 的 App 专用接口实际上没有任何 `x-app-token` 校验，且会为任意传入账户公钥签发人口快照/投票凭证，存在未鉴权签名服务暴露问题
- `sfid/backend` 的投票资格与人口快照人数计算直接依赖本地 `bindings_by_pubkey`，没有以链上状态作为实时唯一真值，和技术文档口径不一致
- SFID 相关技术文档存在实现漂移与乱码残留：`x-app-token` 认证、链上唯一真值表述与当前实现不一致，`SFIDCODEAUTH_TECHNICAL.md` 有乱码
- `voting-engine-system` 的 weights 注释仍引用旧的 `SfidToAccount` 存储名，`sfid-backend` 也有多处 dead_code warning 残留

- 关键结论：
  1. `sfid/backend/src/chain/app_api.rs` 的 `app_voters_count`、`app_vote_credential`、`app_bind_request` 没有任何 `x-app-token` 校验；`main.rs` 也没有给 `/api/v1/app/*` 挂认证中间件。当前实现等价于公开暴露签名接口。
  2. 上述 App 接口只要求传入 `account_pubkey`，不要求证明调用者控制该账户；满足本地绑定/资格条件时，服务端会直接签出链上可用的投票凭证或人口快照凭证。
  3. `chain/vote.rs`、`chain/voters.rs`、`chain/app_api.rs` 的资格判断与 `eligible_total` 统计直接读取本地 `bindings_by_pubkey`，没有实时读取链上绑定关系；这和技术文档“以链上 Runtime 为唯一验签真值 / 投票资格真值”的表述不一致，属于信任边界偏移与一致性风险。
  4. `sfid-code-auth` pallet 本体这轮未发现新的高危漏洞；一对一绑定、防重放、换绑释放、主备轮换约束、领奖联动测试都成立。
  5. 文档与残留问题存在：`SFIDCODEAUTH_TECHNICAL.md` 有乱码，`CHAIN_TECHNICAL.md` 仍写“App routes 使用 x-app-token 认证”的口径，`voting-engine-system/src/weights.rs` 还在引用旧存储 `SfidToAccount`，`sfid-backend` 的 `cargo check/test` 有多条 dead_code warning。
