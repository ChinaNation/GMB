## 实施记录

- 任务卡已创建
- 已读取 `citizenchain/runtime/governance/voting-engine/src/joint_vote.rs`、`internal_vote.rs`、`lib.rs`、`citizenchain/runtime/src/configs/mod.rs` 与 `admins-change/src/lib.rs`。
- 核查确认：联合投票阶段管理员资格与管理员数量来自 `AdminSnapshot`，但机构内部通过阈值在 `do_joint_vote` 中通过 `T::InternalThresholdProvider::pass_threshold(org, institution)` 实时读取。
- 对照确认：内部投票创建时写入 `InternalThresholdSnapshot`，投票时读取该快照；当前没有联合投票专用阈值快照。
- 影响判断：生产 runtime 的阈值提供器读取 `admins-change` 的 Active 主体阈值；当前 `admins-change` 没有直接修改 Active threshold 的公开入口，管理员替换只替换成员不改阈值，联合阶段互斥锁会阻止管理员集合变更提案并发。因此当前更准确地归类为“代码层一致性缺口 / 后续阈值变更路径会触发的问题”，不是已发现可由现有普通 extrinsic 直接触发的阈值篡改路径。

## 核查结论

- 问题存在：`joint_vote.rs` 的联合投票阈值未随联合提案创建快照，而是投票时动态读取。
- 风险条件：只要运维、runtime upgrade、治理扩展或未来模块增加 Active threshold 修改路径，历史联合提案就可能出现管理员名单按旧快照判定、阈值按新状态判定的不一致。
- 建议修复方向：为联合投票增加按 `(proposal_id, institution)` 存储的机构阈值快照，创建联合提案时与管理员快照一起写入，`do_joint_vote` 只读取该快照；同步补测试覆盖“创建后阈值变化不影响既有联合提案”的场景，并更新治理投票文档与权重说明。
