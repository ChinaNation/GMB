## 核查结论

- `InternalAdminProvider` 一致性约束问题存在：trait 仅声明 `is_internal_admin` 与 `get_admin_list`，未明确二者必须来自同一主体版本；内部提案创建先判定发起人权限，再写管理员快照，快照后没有防御性校验发起人仍在快照内。当前 runtime provider 读 `admins-change` 同一套 Active/Pending 主体，现有实现大概率一致，但 trait 层缺契约与防线。
- 已清除：`admins-change` storage version 超前声明已由后续任务调整为 1，本条不再作为待办。
- 已清除：`admins-change` 的 `integrity_test` 已去掉 `std` 编译守卫，wasm/no_std 路径也会纳入完整性断言。
- `admins-change` 的 `ProposalData = MODULE_TAG + action` 与 `ProposalOwner = MODULE_TAG` 双重标记问题存在；并且同类模式在多个治理业务模块中存在。当前 `register_proposal_data` 不会自动加前缀，所以不是当前双前缀故障，但与 `ProposalOwner` 作为唯一归属来源的文档口径不完全一致。
- `voting-engine` 的 expiry retry `expect` 存在低风险 panic 边界：当前容量不变量成立，但 runtime 路径仍可改成 defensive 处理以免未来维护破坏不变量。
- `admins-change` 权重文件明确是保守临时权重，占位问题存在，应重新 benchmark。
- `ExecutionRetryGraceBlocks = ConstU32<21_600>` 与当前 `BlockNumber = u32` 匹配；风险是未来 BlockNumber 类型变化时耦合暴露，不是当前运行错误。
- 已清除：`governance_org_pass_threshold` 兼容旧调用名已删除，调用方已统一到 `fixed_governance_pass_threshold`。
- 已清除：`cleanup_joint_proposal` / `cleanup_internal_proposal` trait 空实现已从 trait 与实现中删除。
- 已清除：`runtime-upgrade::finalize_joint_vote` 手工 Root extrinsic 与 runtime 免费分支已删除，call index 1 保持空缺，终结只走 voting-engine 回调。
- `advance_joint_to_citizen` 只释放互斥锁，不删除联合阶段投票表；这些表会进入 90 天终态延迟清理。行为本身可接受，但文档缺少“公民阶段只读历史快照”的明确说明。
- `admins-change` benchmark 直接写 `voting_engine::Proposals` 和 `ProposalExecutionRetryStates`，跨 pallet 依赖内部状态结构，维护风险存在。
