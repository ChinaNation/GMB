## 审查结论

### 已确认

- 未发现普通外部用户可直接利用的高危权限绕过漏洞
- 三段式状态机、联合回调原子回滚、自动超时重试、90 天延迟清理、SFID 投票凭证防重放这些主路径已实现
- 单测、模块编译、runtime-benchmarks 编译和 runtime 总体编译均通过
- 中文注释总体完整，关键状态机、回调原子性、清理策略与投票资格约束都有说明

### 已确认问题与改进点

1. 功能需求未严格实现：联合提案没有纳入“每机构活跃提案上限”的统一限制。
   - 文档和 `ActiveProposalsByInstitution`/`active_proposal_limit.rs` 都写的是“不区分提案类型”
   - 但代码只在 `do_create_internal_proposal` 调用 `try_add_active_proposal`
   - `do_create_joint_proposal` 没有为 NRC 或其他机构占用活跃提案名额，因此联合提案路径可绕过这条统一限制

2. `weights.rs` 明显失真，且存在低估风险。
   - `create_internal_proposal` 的 weight 只申报了 `NextProposalId + Proposals`，但当前真实路径还会读写 `CurrentProposalYear`、`YearProposalCounter`、`ActiveProposalsByInstitution`、`ProposalsByExpiry`
   - `joint_vote` 的 weight 注释没有反映 `JointVotesByAdmin`、`JointInstitutionTallies`、动态管理员/阈值提供器等实际路径
   - `citizen_vote` 仍引用已删除的 `SfidCodeAuth::SfidToAccount`
   - 当前 weight 不能视为可信

3. benchmark 已与当前 ID/存储模型脱节。
   - `create_internal_proposal` benchmark 仍手工把 `NextProposalId` 置 0，并断言 `Proposals` 里存在 key=0
   - 当前 `allocate_proposal_id` 已改为“年份 × 1,000,000 + 年内计数器”
   - 这说明 benchmark 逻辑本身就没有跟上实现，进一步削弱了 weight 结果可信度

4. 联合转公民阶段时，旧 expiry 桶索引移除写错 key。
   - `schedule_proposal_expiry` 使用的是 `end + 1`
   - 但 `advance_joint_to_citizen` 删除旧索引时用的是 `old_end`
   - 实际效果是旧 expiry 索引不会被主动清掉，只能等后续 `on_initialize` 兜底跳过并自然丢弃

5. 技术文档和残留问题仍有：
   - 文档把 `NextProposalId` 写成简单自增计数器，没有反映当前 `CurrentProposalYear + YearProposalCounter` 模型
   - 文档状态机没有写出 `STATUS_EXECUTION_FAILED = 4` 这类当前代码已支持的联合回调纠偏状态
   - `memory/scripts/load-context.sh` 未登记 `voting-engine`
   - 测试名 `citizen_timeout_cleanup_requires_explicit_joint_cleanup_request` 已与现实现相反，属于历史残留命名

### 说明

- 当前模块最核心的实质问题不是直接安全漏洞，而是“联合提案活跃上限未统一生效”以及“benchmark/weights 已不再可信”
