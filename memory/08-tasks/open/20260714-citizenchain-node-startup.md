# CitizenChain 桌面节点启动异常修复

## 任务目标

- 修复桌面端内嵌节点绕过 CLI 默认交易池配置后异常退出的问题。
- 首次本地数据准备统一显示“初始化中”，已有数据启动统一显示“启动中”，不向用户展示创世过程文案。
- 节点启动阶段若后台必要任务退出，立即结束等待并向前端返回真实退出原因。

## 修改范围

- `citizenchain/node/src/desktop/`：统一桌面节点交易池配置并保留线程退出原因。
- `citizenchain/node/src/home/`：区分首次初始化与普通启动，传播真实错误。
- `citizenchain/node/src/shared/`：RPC 就绪等待支持节点存活检查。
- `citizenchain/node/frontend/home/`：调整生命周期文案并显示真实错误。
- 现有 CitizenChain 节点技术文档、相关 ADR、AI 约束和历史任务卡：统一初始化与普通启动口径。

## 边界

- 不修改 `citizenchain/runtime/`。
- 不修改 chainspec、创世链状态包或现有链数据。
- 不保留旧生命周期状态兼容分支。
- 不执行 GitHub 推送或远端 CI。

## 验收

- Rust 编译检查与前端类型检查通过。
- 使用真实桌面开发程序启动现有本地节点，确认 RPC 可用且节点不会因 `txpool-background` 退出。
- 确认已有数据库启动显示“启动中”，首次数据准备代码路径只显示“初始化中”。
- 人为启动失败时，界面和命令返回真实节点退出原因，不再只显示固定“异常退出”。

## 真实运行发现

- 旧实现固定等待 5 秒后丢弃启动错误接收端，但本机 `new_full()` 约 19 秒才返回，导致真正错误永久丢失。
- 修复启动握手并真实运行后，底层错误已确认是：`节点守卫:创世固定治理骨架校验失败:AdminAccountDecodeFailed([78, 82, 67, 0])`。
- 当前冻结 chainspec / 本地数据库仍使用旧 `AdminAccount` SCALE 模型，当前节点守卫按 fail-closed 拒绝启动。该问题不能通过关闭守卫或兼容旧字段绕过，必须按预上线重新生成链初始状态并一次性替换本机旧链数据。
