## 关键技术点与风险

1. **联合投票快照处理**：决议发行的 extrinsic 需要 `eligible_total + snapshot_nonce + signature`，这三者必须与 `runtime_root_upgrade` 走同一套快照生成路径。开发前先读 `build_propose_runtime_upgrade_sign_request` 对这三个参数的取值逻辑，严格复刻，避免二次发明导致快照不一致。
2. **allocations 上限**：需读 runtime `ResolutionIssuanceMaxAllocations`、`ResolutionIssuanceMaxTotalIssuance`、`ResolutionIssuanceMaxSingleIssuance` 三个常量值，前端必须在提交前本地校验，避免 extrinsic 被链层拒。
3. **验证密钥是内部投票而非联合投票**：不需要快照三件套，只要本机构内部管理员凑票即可。`grandpa_key_gov::propose_replace_grandpa_key` 的 origin 检查走 `is_internal_admin`，走 `InternalVoteEngine::create_internal_proposal`。实现时**不要**错按联合投票处理。
4. **new_key 弱公钥检查**：链上已做 `is_small_order` 拒绝，但前端最好也提示用户，避免白跑一次签名流程。
5. **轻节点 SCALE 编码**：Dart 端没有现成 parity-scale-codec，需要手写编码，参考 `runtime_upgrade_service.dart` 的既有模式。
