## 实施记录

- 任务卡已创建
- 已核对区块链 Runtime、治理入口、SFID 后端输出与模块文档的人口快照契约。
- 结论：当前核心人口快照数据已对齐。
- 对齐点 1：链上验签 payload 为 `(DUOQIAN_DOMAIN, OP_SIGN_POP, genesis_hash, who, eligible_total, snapshot_nonce)`，见 `citizenchain/runtime/src/configs/mod.rs`（2026-04-20 统一 DUOQIAN_V1 方案后）。
- 对齐点 2：SFID 后端签名 payload 与链上完全一致，见 `sfid/backend/chain/runtime_align.rs`。
- 对齐点 3：SFID 对链返回字段为 `genesis_hash`、`who`、`eligible_total`、`snapshot_nonce`、`signature`，见 `sfid/backend/chain/voters.rs`。
- 对齐点 4：链上创建联合提案时实际消费字段为 `eligible_total + snapshot_nonce + signature`，并校验 nonce 不可复用，见 `citizenchain/runtime/governance/voting-engine/src/joint_vote.rs`。
- 对齐点 5：`as_of` 与 `eligible_total` 在 SFID 中于同一次统计窗口生成，符合文档口径。
- 备注：`memory/01-architecture/sfid/README.md` 对人口快照的说明偏概括，写成“仅统计 NORMAL 且绑定有效用户”；当前代码实际直接统计 `bindings_by_pubkey` 中 `CitizenStatus::Normal` 记录。由于该存储本身就是已绑定集合，当前未构成实质不对齐，但后续可考虑把这处文档再写得更精确。
- 进一步扩展核对范围到 SFID -> 区块链全部核心契约后，结论如下：
- 结论分层 1：链上真正消费的核心数据契约已对齐。
- 绑定契约对齐：链上 `bind_sfid(origin, credential)` 实际消费 `{binding_id, bind_nonce, signature}`；SFID 生成的绑定 payload 与 Runtime verifier 一致。
- 投票契约对齐：链上 `citizen_vote(origin, proposal_id, binding_id, nonce, signature, approve)` 实际消费 `binding_id + proposal_id + vote_nonce + signature`；SFID 生成的投票 payload 与 Runtime verifier 一致。
- 人口快照契约对齐：链上联合提案创建实际消费 `eligible_total + snapshot_nonce + signature`，并把发起账户 `who` 纳入验签 payload；SFID 生成人口快照签名与之一致。
- 结论分层 2：如果标准提升为“SFID 对外返回必须严格只有链上需要字段，一个都不多”，当前还不能定义为完全纯净对齐。
- 现状原因 1：绑定结果接口额外返回 `is_bound`、`message`、`key_id`、`key_version`、`alg`、兼容字段 `sfid_signature`。
- 现状原因 2：投票校验接口额外返回 `is_bound`、`has_vote_eligibility`、`message`、`key_id`、`key_version`、`alg`。
- 现状原因 3：旧版人口快照接口额外返回 `as_of` 与兼容结构；现已按最小字段口径收口。
- 以上扩展字段目前不破坏链上核心契约，也与文档说明大体一致；但若后续要建立“严格最小化接口契约”作为功能开发前置条件，则仍建议再做一次接口收口和文档冻结。
- 已补查并确认完整“五项能力”口径，而不再只看 3 条签名链路。
- 功能 4 对齐结论：机构登记能力已切换为 proof 模式。链上真实入口已收口为 `DuoqianManage.register_sfid_institution(sfid_id, register_nonce, signature)`，验签 payload 为 `(DUOQIAN_DOMAIN, OP_SIGN_INST, genesis_hash, sfid_id, name, register_nonce[, signing_province])`（2026-04-20 统一到 `DUOQIAN_V1 + op_tag`，原 `GMB_SFID_INSTITUTION_V1/V2` 彻底退役）；SFID 实际提交时会生成 `genesis_hash + sfid_id + register_nonce + signature` 并一并上链，同时在响应中回写 proof 字段与 `chain_register_tx_hash/block_number`。
- 已在 `memory/05-modules/sfid/backend/chain/CHAIN_TECHNICAL.md` 新增“五项能力总对齐矩阵”，把链上需要字段、SFID 实际提交方式和对齐结论统一冻结。
- 基于最新确认的统一口径再次分析后，影响面分为两类：
