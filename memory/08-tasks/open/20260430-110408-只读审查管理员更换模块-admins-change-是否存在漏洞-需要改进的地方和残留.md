## 审查结论

- 未发现“非管理员可直接发起管理员替换”“跨机构管理员可投票”“已执行提案可重复执行”这类高危直通漏洞。
- 主要风险集中在管理员替换提案与其他提案未互斥、`PASSED` 但未执行的旧提案可在后续状态恢复后被公开重试、Pending 主体读取 API 容易被非投票场景误用，以及 benchmark/weights/文档残留。

## 发现

1. `admins-change` 未实现历史任务卡要求的“管理员更换提案与其他提案互斥”。当前只依赖 `voting-engine` 每机构最多 10 个活跃提案限制；管理员替换提案进入 `PASSED` 后即释放活跃名额，即使自动执行失败也允许新提案继续出现。风险是旧管理员快照下通过的提案可能在后续管理员状态恢复后被公开重试执行，形成跨时期治理语义。
2. `try_execute_replacement_from_action` 只校验投票引擎状态为 `STATUS_PASSED`，未防御性校验 `proposal.kind == INTERNAL`、`proposal.internal_institution == action.institution`、`proposal.internal_org == subject.org`。当前外部用户不能直接写 `ProposalData`，所以不是可直接利用漏洞，但属于跨模块数据误接时的防御缺口。
3. 管理员读取 API 对 `Pending` 主体返回管理员、阈值和人数。这是为了让多签创建提案能锁定管理员快照，但 API 命名没有区分“投票快照可读”和“业务权限 Active 可用”，下游如果只调用 `is_subject_admin` 而未额外校验账户 Active，容易把 Pending 主体误当可执行业务权限。
4. 创世构建写入内置机构管理员时只校验 decode、非空和 `MaxAdminsPerInstitution` 容量，没有在本模块内断言固定人数和管理员唯一性。若 `CHINA_CB / CHINA_CH` 常量未来被改坏，可能导致阈值和实际唯一投票人数不一致。
5. `AdminReplacementVoteSubmitted` 事件已经没有实际发出路径；Phase 2 后投票统一由 `VotingEngine::internal_vote` 发 `InternalVoteCast`，该事件和技术文档中的事件清单属于残留。
6. `src/benchmarks.rs` 仍硬编码 proposal id `0/1`，但当前 `voting-engine` 的 proposal id 是按年份编码分配；benchmark feature 编译能过，但真正跑 benchmark 很可能取不到对应提案。
7. `src/weights.rs` 明确仍是旧实现产物，proof 注释引用已删除的 `ProposalActions / ActiveProposalByInstitution / ProposalCreatedAt / ProposalPassedAt` 等旧存储，计费可信度不足。
8. `memory/05-modules/wuminapp/governance/GOVERNANCE_TECHNICAL.md` 仍列出 `vote_admin_replacement` 等旧业务投票入口，与当前统一 `VotingEngine::internal_vote` 口径不一致。
