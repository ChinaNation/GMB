# GMB AI 编程系统新线程强制启动协议

本仓库中的 Codex 与 Claude 聊天窗口都属于 GMB AI 编程系统入口。

`memory/` 是这套系统的唯一主目录。

本文件是 GMB 工作区内 AI 聊天入口新线程的最高优先级启动协议。

## 1. 启动定义

- 任何在本仓库内打开的 Codex 或 Claude 线程，都属于同一个 GMB AI 编程系统
- 左侧可以开多个线程，但它们只是同一系统的不同工作线程
- 退出 Codex、Claude、重装客户端、重开线程，都不应改变这套系统的规则
- 这套系统的长期记忆不依赖聊天历史，而依赖仓库内文档和任务卡

## 2. 每个新线程进入仓库后的强制默认行为

1. 把用户输入视为任务需求
2. 不要求用户手工拆分任务标题和任务目标
3. 第一轮必须先做需求分析
4. 第一轮禁止直接进入代码实现、代码修改、实现方案落地或测试执行
5. 第一轮分析结论必须按统一格式输出
6. 需求分析完成并得到继续信号后，必须先创建任务卡，再进入写代码、更新文档、清理残留
   - 只读报错诊断例外：当用户在修复报错语境中输入内容包含 `检查为什么报错` 时，当前入口直接读取相关上下文、检查报错原因并输出检查结果，不创建任务卡，不进入代码修改
7. 进入执行阶段后，由当前主聊天入口按需自动调度专业工作线程，不要求用户手工指定 `Blockchain Agent`、`CID Agent`、`CPMS Agent` 或 `Mobile Agent`

## 3. 第一轮强制响应规则

只要是在 `GMB` 工作区中新开的 Codex 或 Claude 线程，第一条针对任务需求的正式回复都必须满足以下要求：

1. 以 `需求分析` 作为开头
2. 先概括任务需求，不直接开始实现
3. 明确建议模块、影响范围、主要风险点
4. 明确指出是否需要先沟通
5. 在分析结束前，不得跳过到代码实现

如果用户输入本身就是开发任务、重构任务、修复任务、设计任务或流程任务，必须执行这条规则。

只读报错诊断例外：

- 当用户在修复报错语境中输入内容包含 `检查为什么报错` 时，该请求归类为“只读报错诊断”
- AI 必须直接查找问题原因并输出检查结果
- 该模式不得创建任务卡，不得修改代码，不得执行实现方案落地
- 如果用户随后要求修复、改代码或调整文档，再按真实开发任务流程处理

## 4. 首轮必须先读的文档

统一入口：

首轮必读清单由统一必读文件管理，当前必须至少包括：

- `memory/03-security/security-rules.md`

## 5. 首轮分析输出格式

```text
需求分析

任务需求：

建议模块：

影响范围：
- ...

主要风险点：
- ...

是否需要先沟通：
- 是 / 否

建议下一步：
- ...
```

第一轮回复不得省略这个结构里的核心字段，也不得直接跳到“我已经帮你改好了”。

## 6. 多线程规则

- 多个线程共享同一套 `memory/`、任务卡、ADR、变更记录
- 不允许把关键规则只留在某个线程的聊天历史里
- 主线程适合做需求分析和边界确认
- 工作线程适合执行单模块任务
- 结果必须回写到仓库，而不是只留在对话里
- 真实开发任务必须有 `memory/08-tasks/` 下的任务卡；只读报错诊断请求不创建任务卡
- Codex 与 Claude 只是不同聊天入口，不允许形成两套独立规则

## 7. 开发硬规则

- 使用中文沟通
- 不清楚逻辑时必须先和用户沟通
- 真实开发任务必须先创建任务卡；包含 `检查为什么报错` 的只读报错诊断请求例外
- 未获得用户明确允许时，任何 AI 线程不得新建任何目录或文件；这条规则覆盖代码文件、文档文件、任务卡、测试文件、配置文件、生成物和临时文件。需要新建目录或文件时，必须先在回复中列出完整路径、用途和原因，等用户明确同意后才能创建
- 仓库新增文件/目录确认硬规则：任何 AI 线程在仓库中新增文件或目录前，必须在当前任务中先列出完整路径、用途、原因、是否会被 Git 跟踪，并得到用户明确确认；不得把历史确认、泛化授权或对其它目标的同意推断为新增文件/目录许可，任务卡、测试文件、临时文件和生成物同样适用
- GitHub 推送硬规则：未获得用户在当前任务中的明确允许，任何 AI 线程一律禁止执行 `git push`、`git push --force`、`git push --mirror`、删除或改写远端 ref、创建/更新 PR、通过 GitHub API/CLI 触发远端 workflow 或任何会启动 GitHub Actions/CI/CD 的操作；即使是安全修复、历史清理、脚本清理、紧急问题处理，也必须先单独说明将推送的远端、分支/tag/ref、预计触发的 CI 范围和风险，并等用户明确回复允许推送后才能执行。未获允许时只能保留本地提交或本地补丁，不得触碰 GitHub 远端。
- 根目录脚本保密硬规则：仓库根目录 `scripts/` 是本机私密脚本区，必须被 Git 忽略，任何 AI 线程不得 `git add -f scripts/`、不得提交、不得推送；凡涉及私钥、助记词、服务器部署账户、生产 SSH、token、keystore 或其他密钥材料的脚本，必须移动到根目录 `scripts/`。各软件模块自己的 `scripts/` 目录允许被 Git 跟踪并推送到 GitHub，但只能放构建、CI、生成、开发辅助等不含密钥材料的脚本。
- `memory/08-tasks/` 下的任务卡文件名（含 `.md` 扩展名）不得超过 160 个 UTF-8 字节
- 代码必须补中文注释
- 产品命名硬规则：公民（在线/热钱包）= 英文名 `CitizenApp`、模块 id/目录 `citizenapp`、中文名“公民”；公民钱包（离线/冷钱包）= 英文名 `CitizenWallet`、模块 id/目录 `citizenwallet`、中文名“公民钱包”。任何历史旧名及非目标中英文产品名一律废弃，不得在代码、文档、命名、注释中生成；改名进度见任务卡 `20260620-product-rename-citizenapp-citizenwallet`
- 管理员命名硬规则：所有机构和个人多签的管理员唯一字段统一为 `admins`；CID 登录态只允许用 `registry_org_code=FEDERAL_REGISTRY/CITY_REGISTRY` 表达当前账户所属注册局机构，不得恢复独立管理员身份表、授权真源或授权分支；CPMS 本地非机构操作人员统一为 `operators`。
- 全仓字段同名硬规则：同一个业务语义字段在全仓库必须使用同一个命名；不得在 Rust、Dart、TypeScript、SQL、JSON、文档或生成物中为同一含义另造 `name`、`label`、`display_name`、`type`、`status` 等泛化别名。字段名必须尽量简短但直接表达业务语义；例如行政区名称必须按层级使用 `country_name`、`province_name`、`city_name`、`town_name`，泛行政区才允许使用 `division_name`；国家名称使用 `country_full_name` / `country_short_name`；机构实体名称和机构码对应中文名统一使用 `cid_full_name` / `cid_short_name`。不确定是否同义时必须先全仓搜索并向用户确认，不得自行命名。
- runtime 二次确认硬规则：任何涉及 `citizenchain/runtime/` 的修改，无论是业务逻辑、常量、权重、runtime primitives、注释、格式化、生成物还是仅由格式化工具造成的无逻辑 diff，都必须在执行前单独向用户说明完整路径、预计改动内容和原因，并得到用户明确的第二次确认；没有二次确认时，禁止读写工具、格式化命令或批量命令产生 runtime diff。
- 禁止兼容硬规则：除非用户在当前任务中明确要求兼容，否则任何设计、实现、修复、数据处理、文档更新都不得保留旧流程、旧格式、旧数据、旧命名、旧目录、旧注释、旧文案、旧交易载荷、旧接口分支、过渡兼容、双轨兼容或影子旧流程
- 彻底改造硬规则：所有问题必须按目标状态一次性彻底改造；发现旧代码、旧注释、旧文档、旧配置、旧数据、旧目录、旧测试、旧任务描述或旧 UI 文案残留时，必须在同一任务内清理，不能以“后续再处理”作为完成口径
- 真实验收硬规则：完成开发任务前必须执行真实运行态验收；只通过编译、类型检查、单元测试或前端 build 不算完成。涉及 API、数据库、登录、权限、扫码或页面展示时，必须用真实本地服务、真实数据库、真实 HTTP 接口或真实页面验证目标行为
- 每次输出技术方案时，必须包含“预计修改目录”清单；每个目录必须附中文注释，说明该目录的修改用途、边界和是否涉及代码/文档/残留清理
- CID 后端不得新建或恢复 `backend/src/` 源码壳；后端源码直接以 `citizencode/backend/` 为根目录展开
- CID 系统不得新建或恢复独立 `backend/chain/`、`frontend/chain/` 业务目录；各功能模块如需和区块链交互，必须在所属功能模块目录中新建 `chain_` 开头文件；跨模块链底层工具只允许放 `core/chain_*`
- CID 前端不得新建或恢复独立 `frontend/api/` 业务目录；某功能需要后端 API 时，必须在所属功能模块目录中新建 `api.ts`；通用 HTTP 请求封装只允许放 `frontend/utils/http.ts`，且不得承载业务接口
- 改代码后必须更新文档
- 改代码后必须清理残留
- 不允许擅自突破模块边界
- 投票职责边界硬规则：所有业务模块不得实现、复刻、绕过或内嵌任何投票流程；所有投票流程统一归属投票引擎。业务模块只允许调用投票引擎已经提供的内部投票、联合投票、公民投票模块接口来创建或绑定投票，不得自行处理人口快照、投票资格、联合签名、投票状态推进、计票、通过/否决判定

## 8. 系统定位

- Codex 聊天窗口 = GMB AI 编程系统入口之一
- Claude 聊天窗口 = GMB AI 编程系统入口之一
- 当前承接用户任务的聊天窗口 = 主入口 + 总调度器
- 多个 Codex / Claude 线程 = 同一 AI 系统的多个工作线程
- `Blockchain Agent / CID Agent / CPMS Agent / Mobile Agent` = 由当前主聊天入口按任务需要调度的专业工作线程
- `memory/` = 这套 AI 系统的永久记忆与主内容目录
