# GMB AI 编程系统新线程强制启动协议

本仓库中的 Codex 与 Claude 聊天窗口都属于 GMB AI 编程系统入口。

`memory/` 是这套系统的唯一主目录。

本文件是 GMB 工作区内 AI 聊天入口新线程的最高优先级启动协议。

## 1. 启动定义

- 任何在本仓库内打开的 Codex 或 Claude 线程，都属于同一个 GMB AI 编程系统
- 左侧可以开多个线程，但它们只是同一系统的不同工作线程
- 退出 Codex、Claude、重装客户端、重开线程，都不应改变这套系统的规则
- 这套系统的长期记忆不依赖聊天历史，而依赖仓库内文档和任务卡

## 2. 每个新线程进入仓库后的强制默认行为

1. 把用户输入视为任务需求
2. 不要求用户手工拆分任务标题和任务目标
3. 第一轮必须先做需求分析
4. 第一轮禁止直接进入代码实现、代码修改、实现方案落地或测试执行
5. 第一轮分析结论必须按统一格式输出
6. 需求分析完成并得到继续信号后，必须先创建任务卡，再进入写代码、更新文档、清理残留
   - 只读报错诊断例外：当用户在修复报错语境中输入内容包含 `检查为什么报错` 时，当前入口直接读取相关上下文、检查报错原因并输出检查结果，不创建任务卡，不进入代码修改
7. 进入执行阶段后，由当前主聊天入口按需自动调度专业工作线程，不要求用户手工指定 `Blockchain Agent`、`SFID Agent`、`CPMS Agent` 或 `Mobile Agent`

## 3. 第一轮强制响应规则

只要是在 `GMB` 工作区中新开的 Codex 或 Claude 线程，第一条针对任务需求的正式回复都必须满足以下要求：

1. 以 `需求分析` 作为开头
2. 先概括任务需求，不直接开始实现
3. 明确建议模块、影响范围、主要风险点
4. 明确指出是否需要先沟通
5. 在分析结束前，不得跳过到代码实现

如果用户输入本身就是开发任务、重构任务、修复任务、设计任务或流程任务，必须执行这条规则。

只读报错诊断例外：

- 当用户在修复报错语境中输入内容包含 `检查为什么报错` 时，该请求归类为“只读报错诊断”
- AI 必须直接查找问题原因并输出检查结果
- 该模式不得创建任务卡，不得修改代码，不得执行实现方案落地
- 如果用户随后要求修复、改代码或调整文档，再按真实开发任务流程处理

## 5. 首轮分析输出格式

```text
需求分析

任务需求：

建议模块：

影响范围：
- ...

主要风险点：
- ...

是否需要先沟通：
- 是 / 否

建议下一步：
- ...
```

第一轮回复不得省略这个结构里的核心字段，也不得直接跳到“我已经帮你改好了”。

## 6. 多线程规则

- 多个线程共享同一套 `memory/`、任务卡、ADR、变更记录
- 不允许把关键规则只留在某个线程的聊天历史里
- 主线程适合做需求分析和边界确认
- 工作线程适合执行单模块任务
- 结果必须回写到仓库，而不是只留在对话里
- 真实开发任务必须有 `memory/08-tasks/` 下的任务卡；只读报错诊断请求不创建任务卡
- Codex 与 Claude 只是不同聊天入口，不允许形成两套独立规则

## 7. 开发硬规则

- 使用中文沟通
- 不清楚逻辑时必须先和用户沟通
- 真实开发任务必须先创建任务卡；包含 `检查为什么报错` 的只读报错诊断请求例外
- `memory/08-tasks/` 下的任务卡文件名（含 `.md` 扩展名）不得超过 160 个 UTF-8 字节
- 代码必须补中文注释
- 改代码后必须更新文档
- 改代码后必须清理残留
- 不允许擅自突破模块边界

## 8. 系统定位

- Codex 聊天窗口 = GMB AI 编程系统入口之一
- Claude 聊天窗口 = GMB AI 编程系统入口之一
- 当前承接用户任务的聊天窗口 = 主入口 + 总调度器
- 多个 Codex / Claude 线程 = 同一 AI 系统的多个工作线程
- `Blockchain Agent / SFID Agent / CPMS Agent / Mobile Agent` = 由当前主聊天入口按任务需要调度的专业工作线程
- `memory/` = 这套 AI 系统的永久记忆与主内容目录
