# GMB 聊天协议

## 1. 对你的输入要求

你在 Codex 或 Claude 聊天窗口里直接输入：

- 中文任务需求

你不需要手工补充：

- 任务标题
- 要交给哪个 Agent
- 具体如何拆分模块

## 2. AI 的默认首轮响应

AI 在第一轮不直接写代码，而是必须先做需求分析。

第一轮正式回复必须：

- 以 `需求分析` 开头
- 先概括任务需求
- 输出建议模块、影响范围、主要风险点、是否需要先沟通、建议下一步
- 不直接进入实现
- 不跳过任务卡直接开工

只读报错诊断例外：

- 当用户在修复报错语境中输入内容包含 `检查为什么报错` 时，AI 直接检查报错原因并输出检查结果
- 该请求不创建任务卡，不修改代码，不执行测试修复，只允许读取代码、文档和错误输出做定位
- 如果检查后用户明确要求修复或改代码，再按真实开发任务流程创建任务卡

标准分析格式见：

- `memory/07-ai/requirement-analysis-template.md`

## 2b. 技术方案输出规则

AI 每次输出技术方案时，必须包含 `预计修改目录` 小节。

`预计修改目录` 小节必须逐项列出本次预计涉及的目录，并用中文注释说明：

- 该目录为什么需要修改
- 修改边界是什么
- 是否涉及代码、文档或残留清理

如果本轮只是只读分析、不进入实现，也必须明确写出“本轮不修改目录”。

## 3. 多线程下的聊天协议

即使你在 Codex 或 Claude 里开多个线程，也要遵守同一套协议：

- 第一轮先分析
- 主线程负责需求分析、模块识别和任务调度
- 工作线程由当前主聊天入口按需分配给 `Blockchain Agent`、`SFID Agent`、`CPMS Agent`、`Mobile Agent`
- 关键结论写回 `memory/` 或任务卡
- 不依赖某个线程私有的聊天历史当长期记忆

## 4. 双聊天入口规则

- Codex 与 Claude 都必须读取同一套 `memory/` 规则
- Codex 与 Claude 都必须先执行需求分析，再创建任务卡，再进入实现
- 包含 `检查为什么报错` 的只读报错诊断请求不进入实现阶段，因此不创建任务卡
- 不允许因为聊天入口不同而形成两套任务模板、两套安全边界或两套文档口径
- 任一入口产出的关键结论，都必须回写到 `memory/` 或任务卡中

## 5. runtime 与扫码签名联动规则

- 当任务涉及 `citizenchain/runtime`，且可能影响 `wuminapp` 在线端或 `wumin` 冷钱包二维码签名/验签兼容性时，AI 必须强制中止单边 runtime 修改
- AI 必须明确提示这是双端联动任务，并先创建跨模块任务卡
- 只有在 `wuminapp` 在线端与 `wumin` 冷钱包的对应更新已纳入本次执行范围后，才允许继续修改 runtime
- 如果本轮只允许分析，不允许直接给出“先改 runtime，双端后补”的建议
